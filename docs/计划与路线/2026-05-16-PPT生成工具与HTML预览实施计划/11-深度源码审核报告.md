## 一、guizang-ppt-skill 源码审核

### 1.1 外部依赖实际情况 [P1 新发现]

计划声称"不照搬外部 CDN 依赖"，但未完整列出需要替代的依赖：

| 依赖 | 来源 | 计划是否提及 |
|------|------|-------------|
| Google Fonts (Inter + JetBrains Mono + Noto Sans SC) | `fonts.googleapis.com` | ❌ 未提及字体替代方案 |
| Lucide 图标 | `unpkg.com/lucide@latest` | ❌ 未提及图标方案 |
| Motion One 动效 | `cdn.jsdelivr.net` (fallback) | ❌ 未提及动效取舍 |

**影响：** 计划说"自包含 CSS/JS"，但没有定义：
- 字体如何自托管或 fallback（Inter/Noto Sans SC 是 Swiss 风格的核心视觉元素）
- 图标用什么替代 Lucide（S04 Six Cells 依赖 `<i data-lucide>`）
- 动效是否首发完全去掉（计划未明确）

**推荐：**
1. 首发 HTML renderer 使用系统字体栈 + 自托管 woff2 子集（Inter Latin + Noto Sans SC 常用 3000 字）
2. 图标首发用 inline SVG path，不依赖 Lucide runtime
3. 动效首发完全去掉，M3 再考虑

---

### 1.2 版式校验器实际复杂度 [P2 新发现]

计划说"复刻 validate-swiss-deck.mjs 的关键约束到 spec 层"。实际校验器有 **7 类规则**：

1. 每页必须有 `data-layout`
2. 版式必须在 S01-S22 白名单内
3. 禁止实验性 P23/P24 结构
4. 非 statement 页禁止 `text-align:center`
5. SVG 内禁止 `<text>`
6. 本地图片必须带 `data-image-slot`
7. S15/S16/S22 图片槽位有比例和裁切规则

**隐藏复杂度：**
- `text-align:center` 检测只扫每页 HTML 的前 1800 字符（`topChunk`），超出范围不检测
- 校验器用全局 regex（`/gi` flag），有状态陷阱

**推荐：** 计划的 Go validator 应覆盖规则 1/2/6/7，规则 3/4/5 在 DeckSpec 层天然不存在（因为不让模型写 HTML）。但规则 7 的图片槽位比例约束需要在 `layouts.json` 中明确定义。

### 1.3 版式隐藏约束 [P2 新发现]

计划未提及的版式级约束：

| 约束 | 影响 |
|------|------|
| Carbon 2x Grid 间距系统（12 档，8px 基线） | PPTX 坐标必须对齐 |
| 中文大标题字号分档（按字符数 4 档降级） | exporter 需要计算字符数决定字号 |
| P0 对齐法则（5 条硬规则） | 违反则整页视觉报废 |
| 版式多样性规则（7-8 页至少 6 个不同 S 编号） | DeckSpec validator 需要跨页检查 |
| `data-image-slot` 系统 | 每张图片需绑定槽位 ID |

**推荐：**
1. 中文标题字号分档规则必须进入 `layouts.json` 的 `pptx_slots` 定义
2. 版式多样性规则首发可以只做 warning（不阻断），M2 升级为 error
3. Carbon 间距系统转换为 PPTX inch 坐标时，建立一个 `spacingToInch(sp int) float64` 工具函数

## 二、html-anything 源码审核

### 2.1 deck.ts 实际用正则而非 DOMParser [P2 新发现]

**计划假设：** "使用 DOMParser.parseFromString"
**实际：** html-anything 的 `deck.ts` **全程用正则**：

```javascript
const SLIDE_RE = /<section\b[^>]*\bclass\s*=\s*["'][^"']*\bslide\b[^"']*["'][^>]*>[\s\S]*?<\/section>/gi;
```

**隐藏陷阱：**
- 全局 regex 有状态，每次调用前必须 `SLIDE_RE.lastIndex = 0`
- `<aside class="notes">` 被正则提取后从渲染 HTML 中删除
- 每页生成的 standalone HTML 注入强制居中样式（`display:flex; align-items:center`）
- `bg` 字段从 section inline style 的 `background(-color)?` 提取，纯 best-effort

**推荐：** 计划已正确决定 Hive 用 DOMParser 而非正则，这是对 html-anything 的改进。但需注意：
1. DOMParser 在 Go 侧不可用（Go 没有浏览器 DOM），Go 测试需要用 `golang.org/x/net/html` 包
2. 前端 `parseDeck.ts` 用 DOMParser 是正确的
3. 不要从 html-anything 复制正则方案

---

### 2.2 deck-viewer 缩略图性能风险 [P2 新发现]

html-anything 的缩略图实现：**每个格子套一个 iframe，1920px 宽然后 CSS scale 到 168px**。

代码注释明确说"最多约 80 张"。30 页 deck = 30 个 iframe 同时渲染。

**计划已正确识别此风险**（4.5.2 说"缩略图首发不要为所有页同时渲染 iframe"），但没有给出替代方案的具体实现。

**推荐：** 首发缩略图用纯文本列表（页码 + 标题前 20 字符），不渲染 iframe。M2 可以加"当前页 ±1 页"的 mini iframe 预览。

---

### 2.3 PPTX 导出的截图前 style hack [P3]

html-anything 的 PPTX 导出在截图前需要**正则替换两处 style**：

```javascript
iframe.srcdoc = slide.html
  .replace(/\.slide\s*\{\s*transform-origin[^}]*\}/, "...")
  .replace(/body\s*\{.*display:flex.*\}/, "...");
```

这段正则极脆弱，依赖 `parseDeck` 注入的 style 字符串格式一字不差。

**推荐：** Hive 的 visual PPTX（M3）不应复制这个方案。服务端 Playwright 截图时应直接控制 viewport 和 CSS，不需要 hack srcdoc。

---

### 2.4 export/image.ts 截图等待策略实际有 5 个阶段 [P2]

计划说"等待字体、图片、layout settle"。实际分 5 个阶段：

| 阶段 | 超时 |
|------|------|
| document readyState | 8s |
| `<link rel="stylesheet">` 逐条等 load/error | 6s |
| `document.fonts.ready` | try/catch |
| `img.complete` + `img.decode()` | 6s |
| Tailwind Play CDN 异步注入 + 2×rAF + sleep(120ms) | — |

**推荐：** Hive 服务端 Playwright 截图（M3）应实现前 4 个阶段，跳过第 5 个（Hive 不用 Tailwind CDN）。首发不需要截图，不阻塞。

---

### 2.5 use-convert.ts 不只是"思路" [P3]

计划说"diff-edit 思路可用于二期"。实际是**完整的 SSE 流式消费实现**：
- 手写 SSE 解析（readline + event/data 拆分）
- per-task AbortController，支持并发多任务
- `asset:<id>` 占位符在发送前内联为 base64 data URL
- Zustand store 按 taskId 隔离状态

**推荐：** 二期"修改这份 PPT"功能不应复制 SSE 方案。Hive 已有 tool call 机制，局部编辑应通过模型修改 DeckSpec 后重新调用 `generate_ppt`，不需要 SSE 流。

## 三、Hive 现有代码集成点审核

### 3.1 AccessResolver 不是链式扩展 [P1 新发现]

**计划假设：** "扩展 presentation 专用 resolver"
**实际：** `ResolveAsset` 方法签名只接受**单个** `AccessResolver`，不是 resolver 链：

```go
func (s *AssetService) ResolveAsset(ctx, uri, rc ResolveContext, resolver AccessResolver, ttl) (string, *AssetRecord, error)
```

如果要为 presentation 加特殊逻辑，必须在 `bootstrap/server.go` 的 `initAssetAccessResolver` 里**包装整个 resolver**，而不是"追加"一个新的。

**推荐：** 在现有 resolver 函数内部加 `source_kind` 分支：

```go
func initAssetAccessResolver(...) AccessResolver {
    return AccessResolverFunc(func(ctx, rec, rc) error {
        if rec.Tags["source_kind"] == "generated_presentation" {
            return canResolvePresentationAsset(ctx, rec, rc)
        }
        // ... 现有逻辑
    })
}
```

这与计划 4.4.1 的设计一致，但计划没有说明这需要修改 `initAssetAccessResolver` 而非"新增"一个 resolver。

---

### 3.2 工具注册应独立函数，不走 variadic [P2 新发现]

**计划假设：** "variadic optional interface 注入依赖"
**实际：** `RegisterBuiltinTools` 的 variadic 槽已有 8 种类型（AgentSpawner、SessionTodoStore、TodoSnapshotBroadcaster、NestedToolGate、PlanRuntimeObserver、TaskBoard、KBService、customerServiceTool）。

`generate_image` 实际是**独立的顶层函数**，在 bootstrap 里单独调用：
```go
tools.RegisterImageGen(sc.MCPHost, sc.AIRouter, logger, cfg.Server.ServerBaseURL())
```

**推荐：** PPT 工具应照搬 `RegisterImageGen` 模式：
```go
// bootstrap/server.go
if sc.AssetService != nil {
    tools.RegisterPPTGen(sc.MCPHost, sc.AssetService, presentationService, logger)
}
```

不要继续扩展 variadic 参数。

---

### 3.3 Asset proxy 当前只支持 inline，不触发下载 [P2 新发现]

**计划假设：** "proxy 支持 Content-Disposition attachment"
**实际：** proxy 端点硬编码 `inline`：

```go
if rec.Filename != "" {
    w.Header().Set("Content-Disposition", fmt.Sprintf("inline; filename=%q", rec.Filename))
}
```

**推荐：** 需要修改 proxy handler，根据 resolve 请求的 `purpose` 或签名参数决定 `inline` vs `attachment`。计划 4.4.2 的设计（HMAC 签名包含 disposition）是正确的，但需要明确这是对现有 proxy handler 的修改，不是新增端点。

---

### 3.4 Canvas 中 `ppt` 类型已存在但只是 Markdown 别名 [P2]

**实际：**
- `ArtifactType` 已包含 `'ppt'`
- `CanvasPanel.tsx` 对 `ppt` 路由到 `MarkdownRenderer`
- `artifactParser.ts` 已在解析白名单中

**推荐：** 计划新增 `'presentation'` 类型是正确的（与 `'ppt'` 区分）。但需要决定：
1. 旧 `'ppt'` 类型是否保留兼容（当前按 Markdown 渲染）
2. 还是迁移为 `'presentation'` 的别名

计划 4.5.1 说"ppt 旧类型不要继续映射为 .md"，但没有给出迁移方案。建议首发保留 `ppt` → Markdown 兼容，新增 `presentation` → DeckRenderer，后续再统一。

---

### 3.5 Bootstrap 初始化顺序约束 [P2]

实际初始化顺序：
```
步骤 7.0：initAssetService（需要 pgPool）
步骤 7.0 后：initAssetAccessResolver（需要 AssetService + KBService）
步骤 11：RegisterBuiltinTools
步骤 11 后：RegisterImageGen / RegisterVideoGen（需要 AIRouter）
```

PPT 工具注册必须在 `sc.AssetService != nil` 之后。如果 `PresentationService` 还依赖 `PresentationRunStore`（需要 pgPool），则初始化顺序为：

```
pgPool → AssetService → PresentationRunStore → PresentationService → RegisterPPTGen
```

**推荐：** 在 bootstrap 中 `initAssetService` 之后、`RegisterBuiltinTools` 之前，加入 presentation 初始化块。

## 七、内置样式与模板策略审核

### 7.1 [P2] 首发 8 个版式覆盖面不足 (confidence: 7/10)

**当前设计：** Swiss 风格，首发 8 个版式（S01/S03/S04/S08/S11/S15/S19/S22）。

**缺口分析：**

| 缺失版式 | 场景 | 用户何时需要 |
|----------|------|-------------|
| S02 Statement（纯大字声明） | 过渡页、章节分隔 | "帮我做一份 10 页汇报"时需要节奏感 |
| S13/S14 KPI Tower | 数据展示 | "季度数据汇报"时核心需求 |
| S21 Closing | 结尾致谢 | 几乎每份 PPT 都需要 |

**推荐：** 首发从 8 个扩到 10 个，补 S02（Statement）和 S21（Closing）。这两个版式结构简单（大字 + 少量文本），PPTX 映射成本极低（各 ~30 行代码）。

---

### 7.2 [P2] 主题色只有 4 个预设，企业适用性受限 (confidence: 8/10)

**当前设计：** 4 个预设色（克莱因蓝 IKB、柠檬黄、柠檬绿、安全橙）。

**问题：** 企业客户品牌色（红色、紫色、深绿等）不在预设中，无法匹配。

**推荐：** 4 预设 + 1 个 `custom` 选项。实现成本极低：
- DeckSpec schema 中 `theme` 枚举加 `"custom"`
- 新增可选字段 `custom_accent: "#hex"`
- HTML renderer 注入 `--accent: var(custom_accent)`
- PPTX exporter 读取 hex 值设置色彩

CSS 变量体系天然支持这个扩展，不需要改动版式骨架。

---

### 7.3 [P1] 字体自托管方案未定义 (confidence: 9/10)

**问题：** Swiss 风格的核心视觉依赖三种字体：
- Inter（西文标题/正文）
- JetBrains Mono（代码/数据）
- Noto Sans SC（中文）

guizang-ppt-skill 通过 Google Fonts CDN 加载。计划说"自包含"但没有定义替代方案。

**影响：**
- HTML 预览：离线/内网环境字体降级为系统默认，视觉质量大幅下降
- PPTX 导出：PptxGenJS 只声明字体名，用户电脑没装 Inter 时 fallback 到 Arial

**推荐：**
1. HTML renderer 内嵌 woff2 子集（Inter Latin ~50KB + Noto Sans SC 常用 3000 字 ~300KB）作为 base64 data URI 或 asset
2. PPTX 字体声明使用 fallback 链：`Inter, -apple-system, Segoe UI, sans-serif`
3. 中文 PPTX 字体声明：`Noto Sans SC, PingFang SC, Microsoft YaHei, sans-serif`
4. 首发 HTML 总大小（含内嵌字体）控制在 500KB 以内

---

### 7.4 [P2] 图标方案未定义 (confidence: 8/10)

**问题：** S04 Six Cells 版式使用 `<i data-lucide="icon-name">` 渲染图标。guizang-ppt-skill 通过 unpkg CDN 加载 Lucide runtime。

**推荐：**
- 首发 HTML renderer：图标用 inline SVG path 直接嵌入（从 Lucide 源码提取常用 20 个图标的 path data）
- 首发 PPTX exporter：图标渲染为小型 SVG → PNG → `slide.addImage`
- 不依赖任何 CDN runtime

---

### 7.5 [P3] 动效首发完全去掉 (confidence: 9/10)

guizang-ppt-skill 每个版式有专属动效 recipe（非统一 fade-up），依赖 Motion One 库。

**推荐：** 首发 HTML renderer 不输出任何动效。理由：
- 计划已决定首发 iframe 不允许 scripts
- 动效在 PPTX 中无法映射
- 去掉动效不影响内容可读性

---

### 7.6 [P2] 中文标题字号分档规则未进入 layouts.json (confidence: 8/10)

**发现：** guizang-ppt-skill 有中文标题字号降级表：

| 字符数 | 字号 |
|--------|------|
| ≤8 | 6.4vw（约 82pt） |
| 9-12 | 5.2vw（约 67pt） |
| 13-18 | 4.6vw（约 59pt） |
| >18 | 必须改写标题或拆行 |

**影响：** 如果 PPTX exporter 不做字号分档，长中文标题会溢出文本框。

**推荐：** 在 `layouts.json` 的每个 text slot 中增加：
```json
"font_sizing": {
  "rules": [
    { "max_chars": 8, "font_pt": 82 },
    { "max_chars": 12, "font_pt": 67 },
    { "max_chars": 18, "font_pt": 59 }
  ],
  "overflow_policy": "reject"
}
```

Node exporter 读取此规则动态设置字号。

---

### 7.7 [P3] 版式多样性规则（跨页检查）(confidence: 7/10)

**发现：** guizang-ppt-skill 要求 7-8 页 deck 至少用 6 个不同 S 编号，不允许连续 3 页同结构。

**推荐：** 首发在 DeckSpec validator 中做 warning（不阻断），M2 升级为 error。实现：遍历 `slides[].layout`，统计去重数和连续重复数。

---

### 7.8 样式策略总结

| 决策点 | 当前计划 | 推荐 | 理由 |
|--------|---------|------|------|
| 首发风格数 | 1（Swiss） | 1 | Swiss 覆盖商务场景 80%+ |
| 首发版式数 | 8 | 10（+S02, S21） | 补过渡页和结尾页，成本极低 |
| 主题色 | 4 预设 | 4 预设 + 1 custom hex | 企业品牌色适配，CSS 变量天然支持 |
| 字体 | 未定义 | 内嵌 woff2 子集 + PPTX fallback 链 | 离线可用 |
| 图标 | 未定义 | inline SVG path（20 个常用） | 不依赖 CDN |
| 动效 | 未明确 | 首发完全去掉 | 无 scripts + PPTX 无法映射 |
| 字号分档 | 未定义 | layouts.json 中定义规则 | 防止中文标题溢出 |
| 二期风格 | Editorial（无时间表） | M3 明确排期 | 用户反馈"风格单一"时有备选 |

**核心判断：** 内置样式方向正确。PPT 生成的价值不是"给用户更多选择"，而是"在有限选择内保证每一份都好看"。但字体、图标、字号分档这些视觉基础设施必须在 Phase 0 定义清楚，否则首发出来的 PPT 视觉质量会远低于 guizang-ppt-skill 的水准。

## 八、LLM/Master 工具执行链路审核

### 7.1 [P0 CRITICAL] 工具结果 20KB 截断机制 (confidence: 10/10)

**发现位置：** `internal/compaction/tool_budget.go`

**当前机制：**
- 单条工具输出阈值：`OutputThreshold = 20KB`（默认）
- 累积上下文预算：`ContextBudget = 40KB`（默认）
- 超阈值的旧工具输出截断为 `[输出已裁剪，原始大小: X KB]`

**致命风险：** 如果 `generate_ppt` 返回 512KB HTML preview（计划允许最大 512KB），**会被压缩管线截断**。LLM 下一轮看到的是占位符，无法基于预览做修改建议。

**推荐：**
1. `generate_ppt` 返回的 `html_preview` 必须从工具结果中移除，改为只返回 `html_asset_uri`
2. 工具结果只包含轻量级结构化 JSON（元数据 + asset URI），控制在 5KB 以内
3. 完整 HTML 通过 asset service 存储，前端通过 `useAsset` hook 拉取

**计划需要修改：** 4.3 节的 `html_preview` 字段设计需要重新考虑。inline HTML 不应进入工具结果。

---

### 7.2 [P0 CRITICAL] 工具执行 30s 超时 (confidence: 9/10)

**发现位置：** `internal/mcphost/client.go`

```go
const mcpOperationTimeout = 30 * time.Second
```

**风险：** MCP 工具默认 30 秒超时。PPTX 生成（DeckSpec 校验 + 图片 resolve + HTML 渲染 + Node worker 启动 + PPTX 生成 + asset 上传）很可能超过 30 秒。

**计划的 `sync_timeout_seconds: 12` 设计是正确的**（在 12s 内完成则同步返回，否则返回 running），但 MCP 层的 30s 硬超时意味着：即使异步模式，工具调用本身（创建 run + 启动 worker）也必须在 30s 内返回。

**推荐：**
1. `generate_ppt` 工具注册时显式设置更长超时（60s）
2. 或者确保工具调用本身只做"创建 run + 返回 running"，不等待 worker 完成
3. 检查 MCP host 是否支持 per-tool timeout 覆盖

---

### 7.3 [P1] 工具 Schema 的 $ref/$defs 透传问题 (confidence: 8/10)

**发现位置：** `internal/llm/tools.go:157-179`

当前实现直接将 `InputSchema` JSON 透传给 `openai.FunctionParameters`，**未做 $ref 解析**。

计划 4.3 节的 tool schema 使用了 `$defs` 和 `$ref`：
```json
"$defs": {
    "DeckSpec": { ... },
    "SlideSpec": { "oneOf": [...] },
    "S01Cover": { "$ref": "#/$defs/S01Cover" }
}
```

**风险：** 部分 LLM provider（特别是非 OpenAI 的国产模型）可能不支持 `$ref`/`$defs`/`oneOf`。

**推荐：**
1. 在工具注册前预处理 schema，将 `$ref` 展开为内联定义
2. 或者使用扁平 schema（每个 layout 独立定义，不用 `oneOf`）
3. 测试覆盖：用实际使用的 LLM provider 验证 schema 解析

---

### 7.4 [P1] tool_choice 机制不会强制调用 generate_ppt (confidence: 7/10)

**发现位置：** `internal/master/tool_choice_detector.go`

当前 `detectToolChoice` 逻辑：
- 纯闲聊 → `none`
- 含 URL/文件路径/skill 引用 → `required`
- 其他 → `auto`

用户说"帮我做一份 PPT"时，没有 URL/路径/skill 引用，返回 `auto`。LLM 可能选择不调工具，直接口头回答。

**推荐：**
1. 在系统 prompt 中明确指示"当用户请求生成 PPT/演示文稿时，必须调用 generate_ppt"
2. 或在 `detectToolChoice` 中增加 PPT 生成意图关键词检测

## 八、前端消息渲染链路审核

### 8.1 [P0 CRITICAL] 工具结果无法自动触发 Canvas (confidence: 10/10)

**发现位置：** `frontend/src/components/chat/MessageBubble.tsx` + `frontend/src/utils/artifactParser.ts`

**当前链路：**
1. `artifactParser.ts` 只解析 **assistant 消息内容**中的 `<artifact>` 标签
2. `role=tool` 消息的 content **完全不经过** artifact 解析
3. 工具结果被"吸收"进 assistant 气泡的 `ToolCallRow → ToolAdapter`，只显示为截断文本
4. 当前对 JSON 工具结果的处理：仅有图片结果（`{url, message}`）有特殊分支，其他 JSON 一律截断显示

**致命问题：** PPT 工具返回 `{kind:"presentation", html_asset_uri:"...", pptx_asset_uri:"..."}` 后，前端**完全不会自动打开 Canvas 或显示预览/下载按钮**。

**计划假设 vs 实际：**
- 计划 4.5 说"ToolResultCard 识别 generate_ppt 的 JSON 输出，显示预览/下载按钮"
- 但当前 `ToolResultCard` 只是 `MessageBubble.tsx` 内部的一个简单函数，对 JSON 只做截断显示
- 没有任何机制从工具结果触发 `useCanvasStore.getState().openArtifact()`

**推荐：** 必须新增以下链路：
1. `ToolResultCard`（或新的 `PresentationResultCard`）检测 `kind === "presentation"`
2. 解析 JSON 后渲染"预览"和"下载 PPTX"按钮
3. "预览"按钮调用 `openArtifact({ type: 'presentation', ... })`
4. 这是**全新的数据流路径**（工具结果 → Canvas），当前代码库中不存在

### 8.2 [P1] MessageBubble.tsx 已 885 行，继续膨胀风险 (confidence: 9/10)

**发现位置：** `frontend/src/components/chat/MessageBubble.tsx`

当前 885 行，内含 8 个子组件全部内联。计划说"首发必须先抽到 ToolResultCard.tsx"，这是正确的，但需要明确：

### 8.3 [P2] Canvas 打开 artifact 的数据来源限制 (confidence: 8/10)

**发现位置：** `frontend/src/components/chat/MessageBubble.tsx:322-343`

当前 Canvas 打开流程：
```typescript
onOpenCanvas={async () => {
```
