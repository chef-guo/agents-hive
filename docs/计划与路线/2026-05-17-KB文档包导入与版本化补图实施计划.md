# KB 文档包导入与版本化补图 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 补齐真实业务知识库导入能力：支持 zip/文件夹中的多 Markdown + 图片相对路径导入，支持已导入文档基于旧版本补图、改 Markdown、发布新版本，并让图片内容通过 Vision insight 进入可检索、可引用的 KB 知识。

**Architecture:** 将“文档包导入”和“单文档补图/新版本编辑”统一为同一套 KB source package pipeline：先解析 source package，规范化 Markdown 文件路径和图片相对路径，做预检，将预检结果写入短 TTL preview cache，最终导入只能基于 `package_id` 复用同一份解析结果。KB 文档新增版本血缘、原始 Markdown、canonical Markdown 三类持久化字段；已发布文档不原地修改，所有编辑以新版本发布，并按 lineage 归档旧 active 版本。batch ingest 的 KB 数据库写入必须全成功或全失败，重建章节树、asset URI rewrite、node asset binding、image insight 视觉索引和 ingest report；对话侧通过 `kb.search` + `kb.section.text` 同时召回文本和图片证据。

**Tech Stack:** Go、PostgreSQL、`archive/zip`、现有 `internal/kb`、现有 `internal/asset`、现有 `internal/fileconv`、统一多模态 LLM 路由计划中的 `AIRouter.ResolveTurnLLM`、现有 `llm.ImageBase64Part` 多模态输入、React + TypeScript + Vite、`FormData` multipart API、现有 `Streamdown` Markdown 预览。

## 0. 当前问题与决策

### 0.1 当前源码事实

当前 KB ingest 已具备这些基础能力：

- `internal/api/kb_handlers.go` 只接受 `multipart/form-data`。
- 单文档预览 API：`POST /api/v1/kb/namespaces/{id}/documents:preview-markdown`。
- 单文档导入 API：`POST /api/v1/kb/namespaces/{id}/documents:ingest-markdown`。
- 单次导入最多一个 `file`，额外图片走重复 `assets` 字段。
- `internal/kb/asset_ingest.go` 会将 Markdown 图片引用重写为 `asset://kb/...`，并按图片语法所在行绑定到章节节点。
- 前端 `frontend/src/pages/admin/KnowledgeBase.tsx` 当前只有单文档导入弹窗，用户可以在导入前编辑 Markdown、选择图片、把图片插入到光标处。
- `internal/kb/types.go` 的 `Document` 当前没有 `parent_document_id` / `lineage_root_id`，不能表达版本血缘。
- `internal/store/postgres_migrate.go` 的 `kb_documents` 当前没有 Markdown 正文字段，不能可靠恢复用户上传的原始 Markdown。
- `internal/api/kb_handlers.go` 当前 multipart 会全量解析请求体，文档包能力首发必须控制并发、响应体大小和缓存总量。
- `internal/kb/tool_section_text.go` 当前只把图片作为 `asset_refs` 返回，包含 `asset_uri/mime/alt/caption/line/page`；没有 OCR、视觉摘要、图表/流程图理解，也不会把图片内容写入可检索文本。
- `internal/master/react_processor.go` 当前图片路由仍停留在入口特例，且会被 `TaskChat` 会话模型覆盖；KB 中已导入的图片不会在召回时自动被视觉模型读取。KB 图片 insight 必须依赖 `2026-05-17-统一多模态LLM路由与输入归一化实施方案.md` 的 turn-level capability route，而不是继续绑定 `TaskVision` 特例。

当前缺口：

- zip 或文件夹中有多篇 Markdown 时没有文档包模型。
- 前端选择图片时只保留 `File.name`，文件夹相对路径会丢失。
- 后端图片解析对 basename 有兜底，遇到多个 `logo.png` 可能错绑。
- 单文档导入成功后没有“基于旧文档新建版本并补图”的入口。
- 没有“预览成功和最终导入必须走同一套路径解析”的强约束。
- 没有跨文档 batch ingest 的原子性承诺；当前 `SaveDocument` / `SaveNodeAssets` 是各自事务。
- 没有可编译约束的 KB package issue code，前后端容易写散字符串。
- 图片在 KB 里只有“位置绑定”，没有“知识化”：Agent 能知道某章节有图，但无法检索图片里的文字、图表、流程、截图含义。

### 0.2 产品决策

1. zip/文件夹导入默认采用 **一篇 Markdown 一个 KB Document**。
2. 多 Markdown 合并为一个 Document 不做首发默认能力；后续可以作为高级选项单独设计。
3. 已导入 Document 不允许原地修改；只能 **基于旧文档新建版本**。
4. 发布新版本后，默认让新版本 `active`，同 lineage 下旧 active 版本自动归档；title 只能作为展示和冲突检测，不能作为版本链依据。
5. 图片路径解析以 Markdown 文件所在目录为 base，不能靠 basename 猜测作为主要逻辑。
6. 找不到本地图片引用时默认阻止发布；允许 warning 导入不是首发默认。
7. 未引用图片默认不上传对象存储，只在报告中提示“未使用图片”。
8. 外链图片默认不下载；保留外链或阻止发布由预检结果给出。首发建议阻止进入 active，避免不可审计外部资产成为 KB 证据。
9. 预览与最终导入必须调用同一套后端 package parser 和 validator，并且最终导入必须使用 preview 返回的 `package_id`，不能重新上传再解析。
10. `kb_documents` 必须同时保存 `markdown_original` 与 `markdown_canonical`：编辑器加载 original，检索和证据使用 canonical。
11. 首次导入的文档 `lineage_root_id = document.id`；新版本写入 `parent_document_id = base.id`，并继承 base 的 `lineage_root_id`。
12. batch ingest 首发采用全成功或全失败的 KB 数据库语义；对象存储上传不是数据库事务的一部分，失败时不得产生可检索 KB 文档，孤儿资产用 `kb_batch_id` 标签和清理任务处理。
13. preview response 默认只返回 summary、文件列表、issue 和轻量资产元数据；Markdown 正文和图片缩略图按需拉取，避免大 JSON 卡死前端。
14. zip symlink、非常规文件类型、路径逃逸、大小写不匹配都按 issue code 明确阻止或提示，不能静默处理。
15. 文档包首发不提升 32MB 请求体限制；如果后续提升到 64MB/128MB，必须先完成流式/临时文件解析或严格全局缓存上限。
16. KB 图片不能只作为附件存在；导入后必须进入“视觉知识索引”流程，至少产出 OCR 文本、视觉摘要、结构化要点、置信度和状态，召回时与章节文本一起返回。
17. 如果没有可用的 `image_input + json/evidence` 路由，文档可以导入，但图片 insight 状态必须是 `skipped_no_multimodal_route`（兼容旧状态 `skipped_no_vision_model` 只用于迁移展示），UI 和召回结果必须明确提示“图片内容未进入可检索知识”；不能假装图片已可问答。
18. 视觉理解默认异步执行；对强依赖图片的知识库，导入时允许 `require_visual_index=true`，图片 insight 全部 ready 前不让文档进入 active。

### 0.3 审核修订结论

Claude Code 审核目录里的 P0/P1/P2 问题已按源码事实复核，以下内容作为本计划的拍板修订，不再按旧契约实施：

| 编号 | 结论 | 计划修订 |
|---|---|---|
| P0-1 版本链缺失 | 成立。`Document.Version` 只是字符串，不能表达血缘。 | 增加 `parent_document_id`、`lineage_root_id`，归档按 lineage，不按 title。 |
| P0-2 preview/ingest 不一致 | 成立。旧计划的 `package_id` 没有被 ingest 使用。 | preview 写缓存，ingest 必须传 `package_id`，不重复上传文档包。 |
| P0-3 Markdown 存储矛盾 | 成立。只存 canonical 会丢原始相对路径。 | 同时存 `markdown_original` 和 `markdown_canonical`，source API 返回两者。 |
| P0-4 batch 部分失败 | 成立。部分成功会制造重试和清理混乱。 | KB 数据库写入全成功或全失败；对象存储孤儿资产不可见并可清理。 |
| P1-1 内存炸弹 | 成立。首发不得盲目提高体积限制。 | package handler 加并发信号量，preview cache 加 TTL、条数和总字节上限。 |
| P1-2 symlink 攻击 | 成立。zip parser 必须拒绝 symlink 和非常规文件。 | Task 3 增加 symlink/special file 测试和检查。 |
| P1-3 同名 active 竞态 | 成立。当前唯一索引拦不住同名不同内容。 | 增加 active title partial unique index；preview 检测同名冲突并要求改名或归档。 |
| P1-4 preview 响应过大 | 成立。不能默认返回所有 markdown 和 base64。 | preview summary 轻量化；新增按需 markdown/thumbnail API。 |
| P2-1 metrics | 采纳，但不引入新 Prometheus 依赖。 | 复用 `internal/observability.MetricsWriter` 写 `hive_metrics`。 |
| P2-2 批量性能 | 采纳为二期优化。 | 首发保证正确性；后续加批量 insert 和并发上传。 |
| P2-3 错误码标准化 | 首发必须做。 | 增加 `IssueCode` 常量和前端 issue 常量映射。 |
| P2-4 并发/性能测试 | 上线前必须补。 | Task 15 增加并发、边界、性能测试。 |
| 新增：KB 图片知识不可召回 | 成立。当前 `asset_refs` 只是引用，不是理解结果。 | 增加视觉 insight 表、异步 vision job、召回返回 image insights 和图片证据 token。 |

## 1. 范围边界

### 1.1 本计划做

- 支持 `.zip` 文档包上传。
- 支持浏览器选择文件夹，保留相对路径。
- 支持一个包内多篇 `.md/.markdown` 和多张图片。
- 严格解析 Markdown 中图片相对路径，包括 `./`、`../`、空格、中文文件名、URL encoded 路径。
- 预检报告展示每篇 Markdown 的标题、行数、图片引用、缺失图片、未使用图片、预计章节数。
- batch ingest：一次包导入创建多篇 KB Document。
- 已导入文档新建版本：拉取当前 Markdown，补图/替换图/移动图片后发布新版本。
- UI 合并为“单文档/文档包/新建版本”三种入口，共用一个 Markdown 编辑与预览面板。
- 日志与报告必须能定位 batch id、每个文档、每张图片绑定结果。
- 图片知识化与召回：导入图片后生成 OCR、视觉摘要、结构化事实、置信度和可检索文本；对话召回时能命中图片内容，并把图片 insight 与图片证据 token 返回给 Agent。
- 新增轻量 `kb.search` 工具，先用现有 PostgreSQL/Store 做文档节点和图片 insight 候选召回；不引入新的向量服务，但不能只靠 Agent 盲看目录猜节点。

### 1.2 本计划不做

- 不做拖拽可视化富文本编辑器。
- 不做 Markdown 合并为单文档的默认能力。
- 不做外链图片自动抓取和远程 URL allowlist。
- 不做每次对话临时读取整包所有原图的在线暴力视觉理解；图片必须在导入或补图后先进入 image insight pipeline。
- 不重做完整 vector/hybrid 检索架构；首发只补图片感知的 `kb.search` 候选召回和 image insight 证据链，后续向量化可复用同一份 `searchable_text`。
- 不改对象存储 provider 架构。

## 3. API 契约

### 3.1 文档包预览

`POST /api/v1/kb/namespaces/{id}/documents:preview-package?domain_id=...&include_markdown=false&include_data_urls=false`

Content-Type: `multipart/form-data`

字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `package` | file | zip 文件。zip 模式使用。 |
| `package_files` | repeated file | 文件夹上传模式使用。每个 part 必须搭配相对路径。 |
| `package_paths` | repeated string | 与 `package_files` 同序，值来自 `webkitRelativePath`。 |
| `mode` | string | `zip` 或 `folder`。 |
| `strict` | bool | 默认 `true`。缺失图片、危险路径、重复 canonical path 都阻止发布。 |

Query:

| 参数 | 默认 | 说明 |
|---|---|---|
| `include_markdown` | `false` | 仅用于小包调试。默认不返回正文，前端通过按需接口读取。 |
| `include_data_urls` | `false` | 仅用于小图调试。默认不返回 base64，前端通过 thumbnail 接口读取。 |

- 缓存 key 与 owner/domain/namespace 绑定，读取时必须重新校验当前用户 scope。
- preview 结果即使含 error issue 也可以缓存，便于前端展示和二次查看。
- ingest 一旦完成 DB materialization 即删除对应 `package_id` 缓存：200 成功、202 视觉索引处理中、409 视觉索引失败且已生成 draft 时都不能依赖 preview cache 重试；后续状态走 `kb_import_batches`。precheck 失败且没有写库时可以保留缓存让用户修复选择。

按需接口：

`GET /api/v1/kb/packages/{package_id}/documents/{index}/markdown`

Response:

```json
{
  "package_id": "kb_pkg_abc123",
  "path": "docs/install.md",
  "markdown_original": "# 安装说明\n\n![架构](../images/arch.png)\n",
  "markdown_lines": 23,
  "image_refs": []
}
```

`GET /api/v1/kb/packages/{package_id}/assets/{index}/thumbnail`

Response:

- `image/png`、`image/jpeg` 或 `image/webp` 二进制响应。
- 服务端将长边压到 240px 以内；无法解码时返回 415 并带 issue code `thumbnail_unsupported`。
- 不返回持久对象存储 URL；这是 preview cache 内临时内容。

### 3.2 文档包导入

`POST /api/v1/kb/namespaces/{id}/documents:ingest-package?domain_id=...`

Content-Type: `multipart/form-data`

字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `package_id` | string | 必填，来自 preview 响应。ingest 不再重复上传 `package` / `package_files`。 |
| `version` | string | 默认 `v1`，应用到所有文档。 |
| `source_uri` | string | 可选，作为包来源。每篇文档会追加包内路径。 |
| `selected_paths` | repeated string | 可选，只导入勾选的 Markdown。为空则导入所有无 error 文档。 |
| `archive_previous_active` | bool | 默认 `true`。新版本按 lineage 归档；全新文档只在同 scope/title 冲突时提示或阻止。 |
| `idempotency_key` | string | 可选，客户端生成。用于网络重试返回同一 batch 结果；不能绕过 `package_id` TTL。 |
| `require_visual_index` | bool | 默认 `false`。为 `true` 时，图片 insight 未全部 ready 前文档不进入 active；用于图片知识强依赖场景。 |

事务语义：

- precheck 阶段不写库，所有 selected documents 必须无 error 才进入写入阶段。
- 写库阶段必须一次性保存 `kb_documents`、`kb_tree_nodes`、`kb_node_assets`、lineage/status 更新；任何一篇失败则回滚所有 KB 数据。
- 对象存储上传无法参与 PostgreSQL 事务；上传的对象必须带 `kb_batch_id`、`kb_package_id`、`kb_document_id`、`source_path` 标签。若后续 DB 回滚，这些对象不可检索，运维可按标签清理。
- 同一个 `idempotency_key` 在同 owner/domain/namespace/package 下重复提交时，返回第一次成功或失败的 batch 结果；不同 payload 复用同 key 返回 409 `idempotency_conflict`。

视觉索引语义：

- 默认导入成功后，图片 insight job 异步排队，文档可 active，但 UI 显示“图片理解处理中”。
- `require_visual_index=true` 时，本次请求走同步等待：DB 先提交 `draft` 文档和 queued insight，worker 或 service 内部执行视觉解析，全部 ready 后同一 lineage 切到 `active` 并归档旧 active。
- `require_visual_index=true` 的等待上限默认 120 秒。超时返回 202 `visual_index_processing`，响应包含 `batch_id` 和文档 draft 状态；前端继续轮询 batch/status，不允许把 draft 当 active 使用。
- 如果多模态路由不可用或任一图片失败，返回 409 `visual_index_failed`；文档保持 `draft`，不归档旧 active。用户可以修复模型能力配置、provider 或图片后重试视觉索引/发布。
- 外部系统通过 API 导入时，如果业务问答强依赖截图/流程图/架构图，建议显式设置 `require_visual_index=true`。

- 同一 lineage 的重复 active：保留 `updated_at` 最新的一条 active，其余 archived。
- 无 lineage 的历史重复：生成运维修复报告，人工确认后归档或改 title；不能自动猜。
- 同一 lineage 下已有重复非空 `version`：迁移前生成冲突报告并要求人工改 version 或归档；不能硬建 `idx_kb_documents_lineage_version`。
- 修复完成后再创建 `idx_kb_documents_active_title`。

### 4.2 版本链规则

- 首次导入：`parent_document_id = NULL`，`lineage_root_id = document.id`。
- 基于旧文档创建新版本：`parent_document_id = base.id`，`lineage_root_id = base.lineage_root_id`；如果 base 没有 lineage，先回填为 base.id。
- 归档旧 active：只能按 `lineage_root_id` 归档，不能按 title 归档。
- title 可以在新版本中变化，版本链不能因此断裂。
- `version` 只做用户可读标签和同 lineage 唯一性检查，不参与排序。
- 同一 lineage 下 `version` 重复返回 409 `document_version_conflict`。
- 全新文档导入时若同 scope/title 已有 active 且不是同 lineage，新文档返回 409 `active_title_conflict`，前端必须让用户改标题或手动归档旧文档。

归档 SQL：

```sql
UPDATE kb_documents
SET status = 'archived', updated_at = NOW()
WHERE domain_id = $1
  AND owner_scope = $2
  AND owner_id = $3
  AND namespace_id = $4
  AND lineage_root_id = $5
  AND status = 'active'
  AND id <> $6;
```

### 4.3 双 Markdown 存储规则

- `markdown_original`：保存用户上传或编辑器提交的原始 Markdown；图片引用保持 `../images/a.png` 这类相对路径。
- `markdown_canonical`：保存 asset rewrite 后的 Markdown；图片引用变为 `asset://kb/...`。
- tree nodes 从 `markdown_canonical` 解析，保证检索和证据引用使用稳定 asset URI。
- 编辑器、新建版本、导出包默认使用 `markdown_original`。
- `kb_node_assets.source_document_path`、`source_asset_path`、`raw_target` 必须保存原始图片引用映射。这样用户基于旧文档新建版本时，即使没有重新上传旧图片，也可以把 `markdown_original` 中仍然存在的相对路径重新解析到旧 `asset://`。
- 若旧历史文档没有 `markdown_original`，source API 返回 409 `document_source_missing`，不允许从 tree node 拼接后让用户编辑。

新版本图片解析规则：

1. 先解析编辑后 Markdown 的图片引用。
2. 如果引用目标是 `asset://`，保持原样。
3. 如果相对路径匹配本次上传的 `asset_paths`，上传新图并生成新 `asset://`。
4. 如果相对路径没有本次上传，但匹配 base document 的 `source_document_path + source_asset_path`，复用旧 `asset_uri` 并写入新文档的 node asset relation。
5. 如果相对路径既不在本次上传，也不在旧映射中，返回 `missing_asset`，阻止发布。

### 4.4 Batch 原子性与对象存储边界

KB 数据库写入必须使用 store 层显式 batch 方法，不能在 service 中循环调用现有 `SaveDocument` 和 `SaveNodeAssets` 后声称原子。

推荐接口：

```go
type PackageDocumentWrite struct {
    Document             Document
    Nodes                []TreeNode
    Assets               []NodeAsset
    ImageInsights        []ImageInsight
    ArchiveLineageRootID string
}

type PackageBatchWriteInput struct {
    BatchID   string
    Scope     ManagementScope
    Documents []PackageDocumentWrite
}

SavePackageBatch(ctx context.Context, input PackageBatchWriteInput) error
```

PostgreSQL 实现要求：

- 一个 `pgx.Tx` 内写入所有 selected documents。
- 每篇文档先写 `kb_documents`，再删除/写入其 `kb_tree_nodes`、`kb_node_assets`、`kb_image_insights`。
- 同一个 tx 内 upsert `kb_import_batches`，记录 idempotency key、summary、issues 和当前 status。
- 在同一个 tx 内执行 lineage 归档。
- 捕获 `idx_kb_documents_active_title` 冲突并转换为 `active_title_conflict`。
- 捕获同 lineage/version 冲突并转换为 `document_version_conflict`。

对象存储上传发生在 DB tx 前或 tx 中都不能真正回滚，因此必须遵守：

- 上传对象必须带 `kb_batch_id` 标签。
- DB 失败后不写任何 `kb_node_assets`，对象不会被 KB 检索到。
- 运维手册提供按 `kb_batch_id` 清理未绑定资产的查询和命令。

### 4.5 图片视觉知识索引

图片导入后必须进入独立的 image insight pipeline，不能只靠 `alt_text` 或 `caption`：

1. `rewriteMarkdownAssets` 上传图片并生成 `NodeAsset`。
2. `SavePackageBatch` 在同一 DB 事务中写入 `kb_node_assets` 和 `kb_image_insights(status='queued')`。
3. `ImageInsightWorker` 读取 queued 记录，通过现有 asset store 根据 `asset_uri` 取回内部对象 key 和二进制，不暴露永久公开 URL。
4. Worker 构造图片 insight 专用 `TurnInputManifest` 和 `TurnRouteRequirement{RequiredCaps: image_input,json}`，调用 `AIRouter.ResolveTurnLLM` 获取 route decision；只在兼容阶段允许 fallback 到 `airouter.TaskVision`，且必须记录 `route_decision_id` 和 fallback reason。
5. 多模态模型输出写入 `ocr_text`、`summary`、`structured`、`searchable_text`、`confidence`、`vision_model`、`route_decision_id`、`status`。
6. `searchable_text` 必须包含：alt、caption、OCR、视觉摘要、结构化要点、所在文档/章节标题。它进入 KB 召回文本，不需要用户显式问“看图”才能被搜索到。

Insight 复用规则：

- 同一 owner/domain/namespace 下 `content_hash` 相同、且旧 insight 为 `ready` 时，新文档版本可以复制旧 OCR/summary/structured/searchable_text，但必须新建本 document/node/asset_uri 的 `kb_image_insights` 行，避免跨文档删除级联误删。
- `asset_uri` 不同但 `content_hash` 相同，evidence token 仍使用新 `asset_uri + content_hash + image_insight_id`。
- 旧 insight 为 `failed/skipped` 时不复制为 ready；新行按当前配置重新排队或标记 skipped。

Vision prompt 固定输出 JSON：

```json
{
  "ocr_text": "图片中的可见文字，保持原语言",
  "summary": "一句话说明图片表达的业务含义",
  "entities": ["产品名", "字段名", "按钮名"],
  "facts": ["可用于问答的事实"],
  "warnings": ["不确定或看不清的内容"],
  "image_type": "screenshot|diagram|flowchart|table|photo|other",
  "confidence": 0.82
}
```

失败/降级规则：

- 无可用多模态路由：写 `skipped_no_multimodal_route`，`searchable_text` 至少包含 alt/caption/source path；旧数据中的 `skipped_no_vision_model` 继续按 skipped 展示。
- 图片过大：生成缩略图/压缩图给 Vision，保留原图 asset；失败写 `failed` 和 error。
- Vision 返回非 JSON：最多重试一次；仍失败则写 `failed`，保存原始错误摘要到 `error`，不能把自由文本硬塞进 `summary`。
- 截图/表格/流程图必须要求模型保留可见文字、按钮名、字段名、步骤顺序；图表数值看不清时进入 `warnings`，不能编造成确定事实。
- OCR/summary 低置信度：仍保存，但召回结果标记 `confidence`，Agent 回答时必须避免过度断言。
- `require_visual_index=true`：只接受 `ready`；`failed/skipped` 都阻止 active 发布。

召回规则：

- `kb.doc.structure` 的节点摘要中应暴露 `image_count`、`image_insight_ready_count`，帮助 Agent 判断该节点是否有图像知识。
- `kb.search` 必须同时检索 tree node 文本和 `kb_image_insights.searchable_text`；图片命中结果应返回 `match_type=image_insight`、`asset_uri`、`node_id`、`summary`、`facts`、`confidence` 和 `evidence_token`。
- `kb.section.text` 返回 `image_insights`，与 `asset_refs` 同级。
- 对于图片 insight 命中的结果，evidence token 必须能指向 `document_id + node_id + asset_uri + content_hash`，而不只是 node text。
- `kb.search` 和 `kb.section.text` 都必须经过 service 层记录当前 turn evidence；store 层只返回候选，不生成 token。
- 如果用户问题明显问“图里/截图/流程图/架构图/表格图片”，工具提示必须引导 Agent 先使用 `kb.search` 找到图片 insight，再用 `kb.section.text` 取同一节点的正文、`asset_refs` 和 `image_insights`。
- 需要二次视觉取证时，后端只生成短 TTL signed URL 或 data URL 给当前 LLM 调用使用；signed URL 不进入持久 KB、不写入聊天 transcript、不返回给前端作为永久链接。

## 5. 核心规则

### 5.1 路径规范化

必须实现为独立纯函数，禁止散落在 handler/UI 中。

输入输出规则：

| 输入 | 当前 Markdown 路径 | 结果 |
|---|---|---|
| `./a.png` | `docs/install.md` | `docs/a.png` |
| `../images/a.png` | `docs/install.md` | `images/a.png` |
| `images/a%20b.png` | `docs/install.md` | `docs/images/a b.png` |
| `/images/a.png` | `docs/install.md` | error: absolute path |
| `../../secret.png` | `docs/a/install.md` | error: path escapes package |
| `asset://...` | any | keep existing asset URI |
| `https://...` | any | issue: remote_url_not_archived |
| `data:image/png;base64,...` | any | inline asset，可进入现有 asset ingest |

大小写策略：

- 首发严格大小写匹配。
- 如果只存在大小写不同的候选，返回 `case_mismatch` error，提示真实路径。
- 不做自动纠正，避免 Linux/macOS/Windows 行为不一致。

### 5.2 Markdown 图片提取

必须跳过 fenced code block：

````markdown
# Doc

```markdown
![不要识别](./fake.png)
```

![要识别](./real.png)
````

必须识别：

- `![alt](path.png)`
- `![alt](path%20with%20space.png)`
- `![alt](<path with space.png>)`

可以不在首发识别 reference-style image：

```markdown
![alt][img1]

[img1]: ./a.png
```

但必须在预检报告中给出 warning：`reference_style_image_unsupported`。不能静默忽略。

### 5.3 图片绑定

导入前后必须一致：

1. preview 解析出每个图片引用的 `document_path + line + raw_target + resolved_path + asset_path`。
2. ingest 使用同一个解析结果或同一套 parser 重算。
3. `rewriteMarkdownAssets` 按 `asset_path` 精确匹配图片二进制，上传到 `internal/asset`。
4. `assignAssetsToNodes` 按 line/page 绑定到最深匹配章节节点。
5. report 中每张图必须显示 bound/unbound、node_id、node_title、line。
6. 每个 bound image 必须创建或复用一条 `kb_image_insights` 记录；没有 insight 状态的图片不能被视为“已完成知识化”。

### 5.4 图片内容召回

Agent 回答 KB 问题时，图片知识进入上下文有三条路径：

- 候选检索：`kb.search` 同时查章节正文、标题/摘要和 `kb_image_insights.searchable_text`，适合用户没有明确文档 ID 或节点 ID 的普通对话入口。
- 文本召回：`searchable_text` 作为章节的扩展知识参与检索和摘要，适合“截图里按钮叫什么”、“流程图步骤是什么”、“架构图有哪些组件”。
- 视觉取证：`kb.section.text` 返回 `image_insights` 和 `asset_refs`。如果 insight 不足或用户明确要求看图，Agent 可以基于短 TTL signed URL 或 data URL 让 vision-capable 模型二次读取原图；这个 URL 只能用于当前模型调用，不能持久化。

### 5.5 新版本发布

新版本必须重新生成：

- `kb_documents` 新记录。
- tree nodes 新记录。
- Markdown content hash。
- node asset relations。
- evidence 后续引用新 document id/version。

旧版本处理：

- `archive_previous_active=true` 时，归档同 namespace + owner + domain + lineage 下当前 active 文档。
- 不按 title 识别旧版本，title 修改不应导致旧版本残留 active。
- 不按 version 字符串判断新旧，因为 `v10`、`2026.05`、`release-a` 不可可靠排序。
- 如果 `version` 与同 lineage 已存在 active 或 archived 文档冲突，返回 409。

## 6. 安全与限制

### 6.1 zip 安全

必须防：

- zip slip：`../`、绝对路径、Windows drive path。
- zip bomb：压缩大小很小但解压后巨大。
- symlink：`f.Mode()&os.ModeSymlink != 0` 必须拒绝，code `symlink_not_allowed`。
- 非普通文件/目录：设备文件、socket、hard link 等必须拒绝，code `unsupported_file_type`。
- 文件数过多。
- 目录层级过深。
- 隐藏系统文件污染：`__MACOSX/`、`.DS_Store`、`Thumbs.db` 默认忽略。
- 可执行文件参与处理：`.exe/.sh/.bat/.dll` 等只记录 ignored，不读入业务数据。

建议首发限制：

| 限制 | 默认值 |
|---|---:|
| 原始请求体 | 首发沿用 32MB |
| zip 文件大小 | 32MB |
| 解压后总大小 | 128MB |
| 文件数 | 1000 |
| Markdown 文件数 | 200 |
| 单篇 Markdown | 2MB |
| 单图片 | 10MB |
| 图片总大小 | 100MB |
| 路径深度 | 20 |

首发不提升当前 `maxKBIngestRequestBytes=32MB`。若后续提升到 64MB，必须同步完成流式/临时文件解析、preview cache 总字节上限、前端文案和压测；不允许前端显示能导入 64MB，后端实际拒绝 32MB。

### 6.2 权限

- package preview/ingest 走 `kbManagementScope`，仅管理者 owner 下操作。
- document source 获取必须校验 owner/domain/document。
- create-version 必须校验旧 document 归属和 namespace 归属。
- `asset://` resolve 仍走现有对象存储 ACL，不能因为管理页 preview 暴露永久 URL。
- package cache 读取必须校验 owner/domain/namespace；不能只凭 `package_id` 返回 Markdown 或缩略图。

### 6.3 并发与缓存限制

首发在 API 层加全局 package semaphore：

```go
type Server struct {
    packageOps chan struct{} // cap 默认 3
}
```

规则：

- preview-package、ingest-package、create-version 中涉及 package parser 或大量 asset rewrite 的路径都必须 acquire。
- acquire 超时 2 秒返回 429 `package_import_busy`。
- cache 中单个 package 的原始内容和派生结构总字节数必须计入全局预算。
- cache 清理 goroutine 每分钟清理过期项；每次写入前也做一次惰性清理。

## 10. 验收标准

### 10.1 功能验收

- zip 文档包可导入多篇 Markdown。
- 浏览器选择文件夹可导入多篇 Markdown。
- 同名图片在不同目录时不会错绑。
- 图片引用缺失时 preview 清楚显示，ingest 阻止发布。
- 未引用图片显示为 unused，不进入对象存储。
- preview 默认不返回 Markdown 全文和 base64 图片；前端按需加载后 Markdown 样式正确渲染。
- ingest 必须使用 preview 返回的 `package_id`；不允许重新上传文档包导入。
- batch ingest 任何一篇 DB 写入失败时，所有 KB document/tree/node asset 写入回滚。
- 已导入文档可以新建版本并补图。
- 新版本发布后旧 active 按 lineage 自动归档，title 修改不影响归档。
- 新旧文档各自的 original source、canonical source、tree、asset bindings 可独立查询。
- 基于旧文档新建版本时，未改动的历史相对图片引用能复用旧 asset，不要求用户重新上传全部旧图。
- 同 scope/title 的两个全新 active 文档不能并发产生。
- 图片导入后必须产生 `kb_image_insights`：ready/failed/skipped 状态可查，UI 可见。
- 可用多模态路由存在时，截图/流程图/表格图片至少生成 OCR、summary、facts、confidence、searchable_text。
- 多模态路由不可用时，状态为 `skipped_no_multimodal_route`，UI 和对话结果明确提示图片内容未进入可检索知识；旧状态 `skipped_no_vision_model` 只作为兼容展示。
- `kb.search` 可以用图片 OCR/summary/facts 命中结果，并返回 `match_type=image_insight`。
- `kb.section.text` 返回同一节点的 `asset_refs` 和 `image_insights`，图片证据 token 指向 `asset_uri + content_hash`。
- `require_visual_index=true` 时，图片 insight 未 ready 的文档不会进入 active，也不会被业务侧对话召回。

### 10.2 安全验收

- zip slip 被拒绝。
- 绝对路径被拒绝。
- 路径 escape package 被拒绝。
- zip symlink 被拒绝。
- zip 非普通文件被拒绝。
- zip bomb 被限制。
- 超文件数/超大小被限制。
- package cache 不能跨 owner/domain/namespace 读取。
- 管理 API 不能读取其他 owner 的 document source。
- `asset://` 不作为公开 URL。
- 短 TTL signed URL 或 data URL 只能用于当前 turn 的 Vision 二次取证，不持久化到 transcript、KB、日志明文。
- 图片 insight 生成必须通过 asset store owner 校验读取对象，不能让模型输入任意外部 URL 或未授权对象 key。

### 10.3 可观测验收

- 每次 package preview/ingest 有 batch id。
- 后台日志能看到解析、预检、cache hit/miss、每篇导入、DB commit/rollback、图片绑定统计。
- UI 导入报告能对应 batch id。
- 失败报告包含文件、行号、原始引用、解析路径。
- `hive_metrics` 能查询 preview/ingest/create_version/image_insight 次数、耗时、失败率、cache 大小、queue depth 和 busy 次数。
- 后台日志能按 `batch_id/document_id/asset_uri` 查到图片 insight queued、processing、ready、failed、skipped 的完整状态变化。

## 11. 风险清单

| 风险 | 缓解 |
|---|---|
| package preview 与 ingest 解析结果不一致 | preview 写缓存，ingest 必须传 `package_id` 并复用缓存；缓存过期必须重新预览。 |
| basename fallback 导致错绑 | strict package mode 禁止 basename fallback。 |
| 已导入文档无法恢复原始 Markdown | P0 补 `markdown_original` 和 `markdown_canonical`；没有 original 的历史文档不开放新版本编辑。 |
| 新版本发布破坏旧 evidence | 不原地修改，旧 document/tree/assets 保留。 |
| 新版本编辑文字时要求重传全部旧图 | `kb_node_assets` 保存 source path 映射，新版本解析时复用旧 asset URI。 |
| title 修改导致旧版本不归档 | 按 lineage 归档，不按 title 归档。 |
| batch 部分成功导致重试重复 | `SavePackageBatch` 单事务；失败返回 zero KB writes。 |
| 对象存储无法随 DB 回滚 | 对象带 `kb_batch_id` 标签，DB 失败后不可检索，运维按标签清理。 |
| preview cache OOM | TTL、最大条数、最大字节、semaphore、默认 32MB 请求体。 |
| preview JSON 过大卡死前端 | 默认不返回 markdown/body data URL；按需加载 markdown 和 thumbnail。 |
| 并发同名 active 竞态 | `idx_kb_documents_active_title` partial unique index + 应用层冲突提示。 |
| symlink 读取包外文件 | zip parser 拒绝 symlink 和非普通文件。 |
| zip bomb | 原始大小、解压后大小、文件数、深度、单文件大小全限制。 |
| 前端选择文件夹丢路径 | 使用 `webkitRelativePath`，同时后端要求 `package_paths` 与文件同序。 |
| 大文件超过当前 32MB 请求限制 | 首发锁定 32MB；是否提升到 64MB 需先做流式/临时文件解析、缓存预算和压测。 |
| 图片只作为 asset ref，无法回答图中内容 | P0 增加 `kb_image_insights`、Vision worker、`kb.search` 和 image evidence；没有 insight 状态的图不能标记为已知识化。 |
| 无多模态路由时用户误以为图片可问答 | 写 `skipped_no_multimodal_route`，UI/召回结果明确提示；`require_visual_index=true` 直接阻止 active 发布。 |
| 多模态模型看错图表数值或截图细节 | 保存 confidence/warnings，低置信度回答必须保守；需要时用短 TTL signed URL 做当前 turn 二次取证。 |
| 图片证据无法审计或换图后误引用旧证据 | evidence token payload 包含 `asset_uri + content_hash + image_insight_id`，`kb_evidence_events` 记录 `evidence_kind=image`。 |
| Markdown reference-style 图片漏处理 | 首发 warning，不静默忽略；后续可实现。 |
