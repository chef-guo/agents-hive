# 统一多模态 LLM 路由与输入归一化实施方案

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task. 本方案是 KB 图片知识化、聊天附件、IM/Embed 多模态输入的共同前置地基；不要在单个入口里继续堆特例。

**Goal:** 让 Hive 在每一轮对话中根据真实输入内容和业务约束，自动选择可处理该轮 `text/image/audio/video/pdf/kb-evidence/tools` 的模型或降级流水线，并把路由、降级、成本和证据全过程可观测化。

**Architecture:** 所有 Web、SDK、IM、Embed 输入先统一进入 asset-backed `TurnInputManifest`，再由 `TurnRequirementBuilder` 计算本轮必需能力，`AIRouter.ResolveTurnLLM` 选择原生多模态模型或辅助服务流水线。模型不能满足能力时，必须按策略自动升级、显式报错或生成可审计降级结果，禁止静默丢图、静默丢视频画面、静默把 PDF 当纯文本。

**Tech Stack:** Go、PostgreSQL、现有 `internal/airouter`、现有 `internal/llm`、现有 `internal/asset`、现有 `internal/fileconv`、现有 `internal/kb`、FFmpeg、React + TypeScript、现有 observability/event bus。

## 0. 行业调研结论（截至 2026-05-17）

### 0.1 OpenAI / Azure OpenAI

官方能力边界：

- OpenAI Images/Vision 文档显示 Responses API 支持 `input_text + input_image`，图片可通过 URL、base64 data URL、Files API file id 传入，并有 `detail` 控制成本与精度。
- OpenAI File inputs 文档显示 `input_file` 可传 PDF；对 PDF，带视觉能力的模型会同时把提取文本和页面图像送入模型；非 PDF 文档默认是文本抽取，电子表格有专门增强流。
- OpenAI Speech-to-text 文档显示 STT 不再只有 `whisper-1`，还有 `gpt-4o-transcribe`、`gpt-4o-mini-transcribe`、`gpt-4o-transcribe-diarize` 等。
- Azure OpenAI 官方 Realtime 文档把实时语音明确拆成 WebRTC、WebSocket、SIP 三类连接；Azure 视觉文档把图片输入定义为文本 + 图片数组，支持 URL 或 base64。

对 Hive 的设计约束：

- `audio` 不能硬绑 Whisper；应走 `ServiceSTT` 或 realtime service 路由。
- `pdf` 不能只用 `ContentFile` 这个粗粒度字段；必须知道当前模型是否支持视觉 PDF，还是只能文本抽取。
- OpenAI/Azure 与其他 provider 的能力不是同一个 API 形状，不能只用 `provider.HasCapability("vision")` 判断。

### 0.2 Anthropic Claude

官方能力边界：

- Claude Vision 文档显示 Claude 能处理图片，API 中图片可以是 base64、URL 或 Files API file id；文档明确建议图片放在文本前面，并说明图片数量、请求大小、缩放、空间定位和计数等限制。
- Claude PDF 文档显示 PDF 支持需要区分直接 API、Vertex、Bedrock。特别是 Bedrock Converse API 中，如果没有启用 citations，可能退化为基础文本抽取，无法看到 PDF 图表。
- Anthropic feature overview 明确把 Files API、PDF support、Citations、Prompt caching 当成独立能力面。

对 Hive 的设计约束：

- 需要 `citations`、`document_visual`、`files_api`、`prompt_cache` 这类 provider-specific capability。
- KB 证据不能只返回文字；图片、PDF 页面、图表必须有 evidence token 和引用状态。

### 0.3 Google Gemini

官方能力边界：

- Gemini Video understanding 文档显示 Gemini 能处理视频，并且同时处理音频流和视觉流；支持 File API、inline video、YouTube URL，能按时间戳提问，默认按 1 FPS 采样，也支持自定义 FPS。
- Gemini Audio understanding 文档显示 Gemini 可以对音频做总结、问答、转录、翻译、说话人区分、情绪分析和时间戳，但实时转录要走 Live API 或 Cloud Speech-to-Text。
- Gemini Document understanding 文档显示 Gemini 对 PDF 是 native vision 文档理解，能理解文本、图片、图表、表格和布局，PDF 限制是 50MB 或 1000 页；非 PDF 文档会作为普通文本处理，视觉布局信息会丢失。

对 Hive 的设计约束：

- `video` 是一等输入模态，不是“抽音频后当文本”。
- 没有原生视频模型时，降级路径必须是 `抽帧 + 音频转录 + timeline summary`，不能只有音频。
- 长视频要有 `fps/media_resolution/duration` 策略，否则成本和延迟不可控。

### 0.4 AWS Bedrock / 国内模型

官方能力边界：

- Bedrock Converse `ContentBlock` 是 union，包含 `text/image/document/video/audio/toolUse/toolResult/reasoningContent` 等块；用户指南说明 image/document/video 都可以传 bytes 或 S3 URI，并提醒 document name 有 prompt injection 风险。
- Bedrock model cards 是按模型列出输入/输出模态、API、feature、区域、quota，不同模型能力差异很大。
- Qwen-Omni / Qwen-VL 方向已经具备 text/image/audio/video 的统一多模态能力，但云 API 仍可能存在“单个 user message 只能组合文本 + 一种其他模态”等具体限制。

对 Hive 的设计约束：

- 需要 provider/model 级 capability override，不能只靠模型名推断。
- 需要 `message_shape` 能力：同一请求是否允许 text+image+video+audio 混合，还是必须拆分为流水线。
- 需要对象存储 URI 作为一等输入，避免大文件在 session JSON 和 SSE 里反复 base64。

### 0.5 2026 平台级最佳实践结论

1. 多模态不是 `vision=true`，而是输入模态、API 形状、文件处理、成本、延迟、证据和降级策略的组合。
2. 本轮路由必须发生在 LLM 调用前，而不是调用失败后补救。
3. 原生多模态优先；无原生能力时才走可审计流水线。
4. 降级必须显式暴露给用户、日志和质量事件，不允许静默省略图片/视频/PDF 图表。
5. KB 图片、PDF 图表、视频片段必须先知识化成 insight，再进入检索证据链；对话时不应临时暴力读取整库原图。

## 1. Hive 当前源码事实

### 1.1 已有基础

- `internal/airouter/types.go` 已有 `ServiceLLM / ServiceImageGen / ServiceVideoGen / ServiceTTS / ServiceSTT / ServiceEmbedding`。
- `internal/airouter/types.go` 已有 `TaskVision`，`internal/airouter/selector.go` 会为 `TaskVision` 选择带 `vision` capability 的模型。
- `internal/llm/content.go` 已有 `ContentText / ContentImage / ContentAudio / ContentFile`。
- `internal/master/prompt_builder.go` 的 `buildUserContent` 可以把图片转为 `llm.ImageBase64Part`，PDF 转为 `ContentFile`，音频/视频转成文本。
- `internal/llm/client.go` 会在调用前校验 provider 是否支持 `vision/audio/file`。
- KB 文档包计划已经把 `image_insight`、`kb.search`、图片证据链列为 P0。

### 1.2 阻塞问题

1. `TaskChat` 仍然返回用户选定/默认主模型，不看本轮输入能力。
2. `internal/master/react_processor.go` 虽然检测图片后设置 `session.activeModel = visionLLM.Model()`，但随后 `prepareDirectExecParams -> getSessionLLM` 会重新用 `TaskChat + session.SelectedModel` 获取 client 并覆盖 `activeModel`，所以图片自动路由并不可靠。
3. `fileconv.Convert` 对 `video/*` 只用 FFmpeg 抽音轨再 Whisper 转录；没有视频画面理解，没有关键帧，没有时间轴。
4. `transcribeAudio` 直接使用 OpenAI Whisper 和 `m.config.APIKey`，绕过 `AIRouter`、DB provider、`ServiceSTT`、审计和成本记录。
5. `ModalityFilterTransformer` 会把不支持的图片/音频替换成文本说明，这能避免 API 报错，但也可能造成用户以为模型看过图片；这必须纳入显式降级事件。
6. `inferCapabilities` 主要来自静态模型表和 provider 粗粒度能力，未读取 `llm_models.config_json` 里的自定义 capability matrix。
7. `ContentFile` 不能表达 PDF 视觉、普通文档文本、视频、asset URI、文件大小、页数、时长、采样策略。

结论：Hive 有多模态零件，但没有平台级多模态路由。

## 2. 拍板设计

### 2.1 不采用的做法

不继续在 `processTaskDirectExec` 写更多 `hasImageAttachments/hasVideoAttachments/hasPDFAttachments` 特例。这个方向会让 Web、IM、Embed、KB 各做一套，最终必然出现某个入口能用、另一个入口静默降级。

### 2.2 采用的做法

新增统一链路：

```text
Web / SDK / IM / Embed / KB
  -> Asset-backed TurnInputManifest
  -> TurnRequirementBuilder
  -> AIRouter.ResolveTurnLLM
  -> Native LLM 或 Auxiliary Pipeline
  -> ReAct / KB Insight / Response
  -> RouteDecision + Evidence + Metrics
```

核心原则：

- 原始输入先入对象存储，session 只保存 `asset_uri + metadata + content summary`。
- 本轮路由基于 `TurnRouteRequirement`，不是基于 `TaskChat`。
- 如果用户选定模型不满足本轮能力，按会话/系统策略处理：
  - `auto_upgrade`：本轮临时切到满足能力的模型，并告知用户/日志。
  - `strict`：直接返回可理解错误，提示用户切换模型。
  - `degrade`：执行可审计降级流水线，比如视频抽帧 + STT。
- 降级结果必须带 `degraded=true`、`reason`、`lost_modalities`，并进入质量事件。

## 3. 新增核心模型

### 3.1 Capability Taxonomy

将现有 `vision/audio/pdf/tools/json` 扩展为可路由的能力矩阵。

必需新增能力：

| Capability | 含义 |
|---|---|
| `text_input` | 文本输入 |
| `image_input` | 原生图片理解 |
| `image_multi` | 单请求多图 |
| `audio_input_native` | 原生音频理解，不只是 STT |
| `audio_transcription` | 可作为 STT provider |
| `audio_realtime` | 实时语音输入/输出 |
| `video_input_native` | 原生视频理解 |
| `video_audio_visual` | 视频同时理解音频流和视觉流 |
| `video_timestamp` | 能按时间戳问答 |
| `pdf_text` | PDF 文本抽取 |
| `pdf_visual` | PDF 页面图像/图表/布局理解 |
| `document_text` | docx/pptx/txt/md 文本抽取 |
| `file_api` | 支持先上传文件再引用 |
| `object_uri_input` | 支持 S3/URL/file id 等对象引用 |
| `tools` | 工具调用 |
| `json` | JSON/结构化输出 |
| `reasoning` | 推理 effort |
| `streaming` | 流式输出 |
| `citations` | 原生引用/文档 citation |
| `prompt_cache` | prompt cache |
| `long_context` | 长上下文 |

### 3.2 TurnInputManifest

所有入口统一转成：

```go
type TurnInputManifest struct {
    SessionID   string
    UserID      string
    Channel     string // web/sdk/embed/feishu/wechat
    Text        string
    Assets      []TurnAsset
    KBEvidence  []KBEvidenceRef
    SourceTrace map[string]string
}

type TurnAsset struct {
    AssetURI       string
    Filename       string
    MimeType       string
    SizeBytes      int64
    Kind           string // image/audio/video/pdf/document/archive
    Width          int
    Height         int
    DurationMs     int64
    PageCount      int
    ContentHash    string
    OriginalSource string
}
```

规则：

- Web 上传、IM 文件、Embed SDK 附件都必须先进入对象存储或临时 upload asset。
- 大文件不进入 `llm.MessageWithTools.Content` 的 base64。
- Manifest 保存足够的 metadata，让路由器不用下载大文件也能决策。

### 3.3 TurnRouteRequirement

```go
type TurnRouteRequirement struct {
    RequiredCaps       []string
    OptionalCaps       []string
    RequiresTools      bool
    RequiresStreaming  bool
    RequiresCitation   bool
    Fidelity           string // low/balanced/high
    LatencyClass       string // realtime/interactive/batch
    CostClass          string // low/balanced/best
    FallbackPolicy     string // auto_upgrade/strict/degrade
    Modalities         []string
    Reasons            []string
}
```

示例：

- 文本 + 图片 + 工具：`image_input + tools + streaming`
- 文本 + 视频：优先 `video_input_native + video_audio_visual`；否则 `audio_transcription + image_input + tools`
- KB 图片问答：`image_insight_ready` 优先；若 insight 缺失，后台异步 `image_input + json`
- PDF 问答：优先 `pdf_visual + citations`；否则 `pdf_text` 并提示图表/扫描页不可见

### 3.4 TurnRouteDecision

```go
type TurnRouteDecision struct {
    DecisionID        string
    Strategy          string // native_model/pipeline/degraded/rejected
    ModelName         string
    ModelID           string
    Provider          string
    CapabilitiesUsed  []string
    AuxServices       []AuxServiceDecision
    Degraded          bool
    DegradeReasons    []string
    UserNotice        string
    EstimatedCost     RouteCostEstimate
    EstimatedLatencyMs int64
}
```

必须写入：

- `hive_logs`
- `agent_quality_events`
- usage/accounting meta
- 前端/SDK 可选 route decision event

## 4. 路由策略

### 4.1 模型选择顺序

1. **Exact native route**：找到同时满足本轮硬能力的 LLM，例如 `text+image+tools+streaming` 或 `text+video+tools`。
2. **Native + auxiliary route**：主模型满足最终对话和工具调用，辅助服务先生成 insight，例如 `STT -> text LLM`、`video native summary -> text/tool LLM`。
3. **Pipeline degraded route**：没有原生视频时，`ffmpeg keyframes + STT + vision summarizer + final LLM`。
4. **Reject with actionable error**：无法保证正确性时拒绝，例如用户问“视频画面里表格第二列是什么”，但没有视频原生模型，也没有 vision 模型。

### 4.2 用户选定模型处理

用户选模型不等于锁死本轮模型。新增三种策略：

| 策略 | 默认场景 | 行为 |
|---|---|---|
| `auto_upgrade` | Web/SDK/Embed 默认 | 本轮临时切到满足能力模型，UI 显示“已因图片/视频切换模型” |
| `strict` | 高合规/评测/调试 | 选定模型不支持即失败 |
| `degrade` | 成本优先/低配部署 | 用 STT、抽帧、文本抽取降级，明确告知损失 |

### 4.3 视频策略

视频必须区分三种问题：

- 用户问视频声音：可以只走 STT/audio understanding。
- 用户问视频画面：必须走 native video 或抽帧 + vision。
- 用户问音画关系：优先 native video；fallback 必须生成 timeline，把 transcript segment 和 keyframe timestamp 对齐。

降级流水线：

```text
video asset
  -> ffprobe metadata
  -> transcript via ServiceSTT
  -> keyframes via ffmpeg scene/fps policy
  -> image summaries via vision model
  -> timeline fusion summary
  -> final ReAct LLM
```

最低要求：

- 静音视频不能失败为“无音频”；必须继续抽关键帧。
- 无画面音频文件不能走视频 pipeline。
- 快速变化视频必须允许提高 FPS，但要受成本和最大帧数限制。

### 4.4 PDF/文档策略

- PDF：优先 `pdf_visual`，因为图表/流程图/扫描页是业务知识的一部分。
- PDF fallback：`fileconv` 提取文本 + 页面图片 insight，必须标记 `degraded_pdf_visual=false/true`。
- DOCX/PPTX/XLSX/MD/TXT：默认文本化；如果文档内嵌图片需要进入 KB，则走 KB 文档包 image insight。

### 4.5 KB 策略

KB 图片、PDF 页面图、视频片段都不能在对话时临时瞎看。导入/补图时必须先生成 insight：

- `ocr_text`
- `visual_summary`
- `structured_facts`
- `layout_notes`
- `confidence`
- `asset_uri`
- `node_id`
- `evidence_token`
- `model/provider/route_decision_id`

对话时：

```text
用户问题
  -> kb.search
  -> 命中文本 node 或 image/video insight
  -> kb.section.text
  -> 带 evidence_token 回答
```

如果 insight 状态是 `queued/processing/failed/skipped_no_model`，Agent 必须说明图片/视频尚未完成可检索解析，不能编造。

## 6. 分阶段实施优先级

### P0-A：模型能力矩阵与路由决策地基

目标：先让系统能正确判断“这轮需要什么能力、哪些模型满足”。

任务：

- 新增 `Capability` 常量和 `TurnRouteRequirement`。
- 解析 `llm_models.config_json.capabilities/limits/provider_features`。
- 实现 `ResolveTurnLLM`：
  - user-selected model 满足时优先用。
  - 不满足时按 `auto_upgrade/strict/degrade` 决策。
  - route decision 必须包含原因和降级信息。
- 单测覆盖：
  - 文本 only 仍走用户模型。
  - 图片 + 用户选 DeepSeek 时自动升级到 vision model。
  - strict 模式下返回 `model_capability_mismatch`。
  - 没有 vision model 时返回 degraded/rejected。

验收：

- 后端日志能看到 `route_decision_id`、required caps、selected model、fallback policy。
- 不再通过写 `session.activeModel` 这种副作用实现图片路由。

### P0-B：Asset-backed TurnInputManifest

目标：所有入口以同一份 manifest 决策，避免 Web/IM/Embed 行为不一致。

任务：

- 将 pending attachments 转为 `TurnInputManifest`。
- 对 asset 做 MIME sniff、hash、size、图片尺寸、音视频时长、PDF 页数探测。
- session message metadata 保存 manifest summary，不长期保存大 base64。
- Web/IM/SDK 入口都能携带 `asset_uri`。

验收：

- 同一张图片从 Web 上传、Feishu 转发、Embed SDK 传入，构建出的 manifest 等价。
- 超大文件在 LLM 调用前失败，错误可读。

### P0-C：图片与 PDF 原生路由修复

目标：修复当前图片自动路由不可靠的问题，并把 PDF 视觉能力纳入 route。

任务：

- `processTaskDirectExec` 首轮在构建消息后调用 `BuildTurnRequirement + ResolveTurnLLM`。
- ReAct loop 使用 route decision client，而不是 `getSessionLLM(TaskChat)`。
- `llm.ContentFile` 增加 `MimeType/Filename/AssetURI/FileKind`。
- PDF route：
  - `pdf_visual` 模型：直接传 PDF。
  - 无 `pdf_visual`：走 fileconv 文本提取并标记降级。

验收：

- 用户选择非 vision 模型并上传图片时，本轮确实调用 vision 模型，日志和 UI 都可见。
- PDF 中只有图表的问题，在没有 `pdf_visual` 时不会假装回答。

### P0-D：STT 路由替代固定 Whisper

目标：音频转录纳入 AIRouter、配置、观测和成本。

任务：

- `Master.transcribeAudio` 改为调用 `airouter.Execute(ServiceSTT)` 或新的 `media.AudioPipeline`。
- `adapter_stt.go` 支持 provider/model 配置读取，默认兼容现有 OpenAI Whisper。
- 支持 `gpt-4o-transcribe`、diarization capability 标记。
- 记录 transcript asset、model、duration、confidence/segments。

验收：

- 后台日志能看到 STT provider/model，而不是只看到 Whisper。
- 没有 STT provider 时，音频输入按 policy 明确失败或降级。

### P0-E：视频原生与降级流水线

目标：把视频从“音频转录附件”升级为一等输入模态。

任务：

- 新增 video metadata probe。
- 若存在 `video_input_native + video_audio_visual` 模型，优先原生传视频。
- 否则执行 fallback：
  - 抽音频 -> STT。
  - 抽关键帧 -> vision summary。
  - transcript + frame summary -> timeline fusion。
- 输出 `VideoInsight`：
  - `transcript`
  - `keyframes`
  - `events`
  - `timestamps`
  - `degraded`
  - `lost_details`

验收：

- 静音视频也能通过关键帧生成视觉摘要。
- 用户问“视频 00:10 发生了什么”时，原生模型优先；fallback 至少能按最近 keyframe/timestamp 回答并标记精度。
- 没有 vision/STT 时不编造视频内容。

### P0-F：KB 图片/视频/PDF insight 对齐

目标：KB 文档包计划使用同一套多模态路由，不再直接依赖 `TaskVision`。

任务：

- KB `image_insight_worker` 调用 `ResolveTurnLLM`。
- 图片 insight 失败状态区分：
  - `skipped_no_model`
  - `failed_provider`
  - `failed_safety`
  - `degraded_low_detail`
- 对 PDF 页面图、PPT 图片、文档包图片统一生成 insight。
- `kb.search` 返回 insight 的 route decision 和 evidence token。

验收：

- KB 图片问答必须引用 `evidence_token`。
- 如果图片 insight 没 ready，Agent 必须说明无法读取图片内容。

### P1：Provider API shape 完整适配

目标：让路由结果能稳定序列化到 OpenAI Responses、OpenAI Chat、Anthropic Messages、Gemini generateContent、Bedrock Converse。

任务：

- 增加 provider serializer abstraction。
- 支持 Files API / asset URL / S3 URI / base64 的选择策略。
- 对 Anthropic Bedrock PDF citations caveat 加 provider feature。
- 对 Gemini video 的 File API processing、FPS、media resolution 加参数。

验收：

- 同一 `TurnInputManifest` 可以在不同 provider 下生成符合各自 API 的请求。
- 不支持混合模态的 provider 会被路由器提前排除或拆 pipeline。

### P1：前端管理与用户可见反馈

目标：管理员能配置能力，用户能知道本轮为什么换模型或降级。

任务：

- LLM 管理页增加 capability matrix。
- Chat 页面展示 route notice。
- 附件上传后预估“需要视觉/视频模型”。
- 错误/降级文案国际化。

验收：

- 管理员可以把某模型标记为 video native 或禁用 image。
- 用户不会误以为模型看过被降级省略的图片/视频。

### P2：成本、缓存与质量评测

目标：把多模态能力从可用做到可运营。

任务：

- 图片 detail/FPS/media_resolution 自动策略。
- 视频/图片 insight cache by content hash。
- route decision replay benchmark。
- 每日质量评测覆盖图片、PDF、视频、音频、KB evidence。

验收：

- 同一图片重复导入不会重复花 vision token。
- 大视频不会无限抽帧。
- 质量报告能看到多模态失败原因分布。

## 7. 与其他计划的协同

### 7.1 统一对象存储

本计划依赖统一对象存储。多模态输入不能长期存在 session JSON 里，必须先获得 `asset_uri`，再由 router 决定是传 file id、URL、S3 URI、base64，还是先做本地预处理。

### 7.2 KB 文档包导入与版本化补图

KB 计划中的 `image_insight_worker`、`kb.search`、PDF 图表理解必须依赖本计划的 P0-A/P0-C/P0-F。

建议调整 KB 计划：

- 将 `现有 internal/airouter TaskVision` 改为 `Turn-level multimodal route decision`。
- `require_visual_index=true` 时，不是简单找 Vision 模型，而是要求 `image_input + json + evidence`。
- PDF/PPT 中图片 insight 使用同一 `InsightPipeline`。

### 7.3 JS Widget / Embed SDK

Embed SDK 传来的业务上下文和附件必须进入同一 `TurnInputManifest`。外部系统只需要按协议上传资产或传 `asset_uri`，Hive 自己负责模型路由。

对外承诺：

- 业务侧不需要知道该用哪个 LLM。
- 业务侧需要声明文件 MIME、文件名、业务对象 ID、可选权限域。
- Hive 返回 route notice、降级原因和证据引用。

### 7.4 IM 渠道

微信/飞书文件、图片、语音、视频都应先落 asset，再进 manifest。

特别规则：

- 语音消息默认 STT。
- 图片消息默认 vision route。
- 视频消息默认 video route 或 fallback pipeline。
- IM 端如果不能展示富提示，至少在后台日志和 Web 会话详情中可见。

## 8. 风险与治理

| 风险 | 后果 | 治理 |
|---|---|---|
| 静默降级 | 用户以为模型看过图片/视频，实际没有 | 所有降级必须进入 route decision 和用户 notice |
| 大文件成本失控 | 视频/PDF 消耗大量 token | 限制 duration/pages/FPS/detail/media_resolution |
| provider 能力误配 | 路由到不支持模型导致失败 | capability smoke test + 管理页校验 |
| prompt injection via filename/document name | 文件名被模型当指令 | 文档名中立化，原始文件名只进 metadata |
| KB insight 过期 | 图片更新后旧 insight 被召回 | insight 绑定 content_hash 和 document_version |
| 证据不可追溯 | Agent 引用图片但无法审计 | evidence token 绑定 asset_uri/node_id/model/route_decision |
| IM/Embed 权限泄漏 | 外部资产跨用户可见 | asset metadata 校验 owner/domain/session binding |

## 9. 验收标准

P0 完成后必须通过这些场景：

1. 用户选择 DeepSeek 文本模型，上传图片问“图里是什么”；系统本轮自动切到 vision 模型，UI 和日志显示切换原因。
2. 用户选择 strict 模式，上传图片但所选模型不支持 vision；系统拒绝并提示可选模型，不调用 LLM。
3. 用户上传静音视频问“画面里流程是什么”；系统抽关键帧并调用 vision，不再报“音频为空”。
4. 用户上传带声音视频问“00:10 发生了什么”；有 native video 模型时直接调用；没有时走 keyframe + transcript timeline，并标明降级。
5. 用户上传 PDF 问图表问题；没有 `pdf_visual` 时系统提示图表不可可靠解析，不编造。
6. 用户上传音频；后台日志显示 STT provider/model/usage，不再固定 Whisper。
7. KB 导入图片后，图片 insight ready 前对话不能编造图片内容；ready 后可通过 `kb.search` 命中图片事实并引用 evidence token。
8. Embed SDK 传 `text + asset_uri(image)` 与 Web 上传图片行为一致。
9. 所有 route decision 都能按 `session_id/turn_id/route_decision_id` 检索。

## 10. 立即拍板的优先级

1. **先做 P0-A + P0-C**：修复当前图片路由不可靠，这是最小闭环。
2. **并行做 P0-B**：没有 manifest，IM/Embed/KB 会继续分叉。
3. **然后做 P0-D**：把音频从固定 Whisper 收回 AIRouter。
4. **再做 P0-E**：视频一等化，避免业务视频场景继续假支持。
5. **最后接 P0-F**：KB 图片 insight 用统一路由落地。

这套顺序比直接写 KB image worker 更稳，因为 KB image worker 依赖可解释、可配置、可审计的 vision route；否则 KB 会把同样的路由债再实现一遍。
