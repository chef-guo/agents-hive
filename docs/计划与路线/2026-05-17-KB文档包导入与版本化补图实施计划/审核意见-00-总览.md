# KB 文档包导入与版本化补图 - 深度技术审核

> 审核人：技术架构组
> 审核日期：2026-05-17
> 审核方法：逐行阅读计划文档 + 源码交叉验证

## 审核结论

**不建议直接按当前计划实施。** 存在 4 个 P0 阻塞问题需要先修复设计，否则上线后会出现数据不一致、版本链断裂、重试重复导入等生产事故。

## 问题分级统计

| 级别 | 数量 | 含义 |
|------|------|------|
| P0 阻塞 | 4 | 不修复不能上线，涉及数据模型和一致性 |
| P1 重要 | 4 | 影响生产可用性，需在首发前修复 |
| P2 改进 | 4 | 可后续迭代，不阻塞首发 |

## 源码验证基础

以下审核意见均基于源码实际状态，非推测：

- `internal/kb/types.go` — Document 结构体无版本链字段（已验证）
- `internal/kb/pg_store.go` — 有事务但并发防护存在逻辑缺口（已验证）
- `internal/kb/asset_ingest.go` — 确实存在 basename fallback 三级查找（已验证）
- `internal/api/kb_handlers.go` — multipart 全量 `io.ReadAll` 加载到内存（已验证）
- `internal/store/postgres_migrate.go` — `kb_documents` 表无 markdown 正文字段（已验证）

## 做得好的地方

1. 路径解析设计为独立纯函数，可测试性强
2. 不原地修改文档，符合审计要求
3. zip 安全限制考虑全面（zip slip、zip bomb、文件数）
4. 任务拆分清晰，每个 task 有测试步骤
5. 产品决策明确，边界清晰
