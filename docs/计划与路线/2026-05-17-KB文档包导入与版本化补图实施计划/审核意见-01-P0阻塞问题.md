# P0 阻塞问题（不修复不能上线）

---

## P0-1：版本链缺失 — 数据模型设计缺陷

### 源码事实

`internal/kb/types.go` 中 `Document` 结构体（第 40-56 行）完整字段：

```
ID, NamespaceID, DomainID, OwnerScope, OwnerID, SourceURI,
Title, Description, ContentHash, Version, Status,
EffectiveAt, ExpiresAt, CreatedAt, UpdatedAt
```

`Version` 是普通 `string`，仅用于标记版本号（如 "v1"）。**没有 `parent_document_id`、`supersedes_id` 或任何版本链字段。**

### 问题分析

计划中归档策略（Task 9 Step 3）：

```sql
UPDATE kb_documents
SET status = 'archived'
WHERE namespace_id = $4 AND title = $5 AND status = 'active' AND id <> $6;
```

这个策略依赖 `title` 相同来识别"同一文档的旧版本"。存在以下问题：

1. **title 被修改时版本链断裂**：用户在 v2 中把"安装说明"改为"安装指南"，v1 不会被归档，出现两个 active 文档。
2. **无法追溯血缘**：无法知道 v3 是从 v2 派生的，还是全新文档恰好同名。
3. **审计场景失效**：合规审计要求能重建完整版本演进历史，当前模型无法支持。
4. **version 字符串不可排序**：计划中已承认 `v10`、`2026.05`、`release-a` 不可可靠排序，但没有给出替代方案。

### 修复方案

**方案 A（推荐）：增加版本链字段**

```sql
ALTER TABLE kb_documents
  ADD COLUMN parent_document_id TEXT REFERENCES kb_documents(id),
  ADD COLUMN lineage_root_id TEXT;  -- 指向版本链的根文档

CREATE INDEX idx_kb_documents_lineage ON kb_documents(lineage_root_id, created_at);
```

Go 类型：

```go
type Document struct {
    // ... 现有字段 ...
    ParentDocumentID *string `json:"parent_document_id,omitempty"`
    LineageRootID    *string `json:"lineage_root_id,omitempty"`
}
```

归档逻辑改为按 lineage 而非 title：

```sql
UPDATE kb_documents
SET status = 'archived', updated_at = NOW()
WHERE lineage_root_id = $1 AND status = 'active' AND id <> $2;
```

**方案 B（最小改动）**：至少增加 `parent_document_id`，lineage 通过递归查询重建。成本低但查询性能差。

### 影响范围

- `internal/kb/types.go` — Document 结构体
- `internal/store/postgres_migrate.go` — 新增字段和索引
- `internal/kb/pg_store.go` — SaveDocument、归档逻辑
- `internal/api/kb_handlers.go` — create-version 响应需返回 parent_document_id

## P0-2：Preview 与 Ingest 解析结果不一致 — 幂等性缺失

### 源码事实

当前 `handleKBPreviewMarkdown` 和 `handleKBIngestMarkdown` 是两个独立 handler，各自独立解析 multipart。计划中 preview 返回 `package_id`，但 ingest 接口定义（Task 7 Step 4）没有接收 `package_id` 的字段，意味着 ingest 时会重新解析上传的文件。

### 问题分析

1. **时间窗口内的不一致**：用户 preview 看到"0 个错误"，然后在 5 秒内修改了本地文件，再上传 ingest，实际导入的是修改后的文件。这不是 bug，但用户体验是"预览通过了，导入结果却不同"。

2. **`package_id` 是空壳**：计划 API 响应中有 `package_id: "kb_pkg_abc123"`，但 ingest 请求字段列表里没有这个参数。这个 ID 没有任何作用，会误导实现者以为 preview 结果被缓存复用了。

3. **重复解析浪费**：大文档包（200 个文件）preview 一次、ingest 一次，解析了两遍，内存峰值翻倍。

4. **无法实现"预览锁定"语义**：企业场景下，管理员 preview 确认后，希望 ingest 的就是 preview 看到的那批文件，不允许中途替换。当前设计无法保证。

### 修复方案

**方案 A（推荐）：服务端缓存 preview 结果**

```go
// preview 时缓存解析结果，TTL 5 分钟
type PackagePreviewCache struct {
    mu    sync.Mutex
    items map[string]*cachedPreview
}

type cachedPreview struct {
    PackageID string
    Plan      *SourcePackagePreviewPlan
    ExpiresAt time.Time
}

// ingest 时必须传 package_id，复用缓存
type IngestPackageRequest struct {
    PackageID             string   `form:"package_id"` // 必填
    Version               string   `form:"version"`
    SelectedPaths         []string `form:"selected_paths"`
    ArchivePreviousActive bool     `form:"archive_previous_active"`
    // 不再重复上传文件
}
```

**方案 B（最小改动）**：保持当前两次上传，但在 ingest 时对文件内容做 checksum 校验，与 preview 时记录的 checksum 比对，不一致则拒绝并提示"文件已变更，请重新预览"。

### 影响范围

- `internal/api/kb_handlers.go` — preview handler 写缓存，ingest handler 读缓存
- `internal/api/routes.go` — 可能需要注入缓存依赖
- API 契约（Section 3.2）— ingest 请求字段需增加 `package_id`

## P0-3：Canonical Markdown 存储矛盾 — 原始路径信息丢失

### 源码事实

`postgres_migrate.go` 中 `kb_documents` 表（第 1586-1602 行）无任何正文存储列。文档内容被解析为 `kb_tree_nodes` 中的节点，每个节点有 `text` 字段，但这是章节片段，不是完整 markdown。

计划 Section 3.3 注意事项已承认：

> "不能从 tree node text 拼接成源文档后给用户编辑，因为会丢失原始顺序、空行、注释和图片语法上下文。"

### 问题分析

计划 Task 8 要求存储 canonical markdown，但存储的是**重写后**的 markdown（`asset://` URI）：

```markdown
# 安装说明
![架构](asset://kb/user/u1/ns-1/doc_1/abc123.png)
```

原始 markdown 是：

```markdown
# 安装说明
![架构](../images/arch.png)
```

**原始相对路径 `../images/arch.png` 已永久丢失。**

这导致：

1. **新版本编辑体验差**：用户打开"新建版本"编辑器，看到的是 `asset://` URI，不是人类可读的路径。
2. **对象存储迁移困难**：如果 `asset://` scheme 或路径规则变化，无法从存储的 markdown 重建原始引用。
3. **无法导出还原**：用户希望把 KB 文档导出为原始 zip 包，但原始路径已丢失，无法还原目录结构。

### 修复方案

**同时存储两份 markdown：**

```sql
ALTER TABLE kb_documents
  ADD COLUMN markdown_original  TEXT NOT NULL DEFAULT '',
  ADD COLUMN markdown_canonical TEXT NOT NULL DEFAULT '';
```

- `markdown_original`：保存用户上传的原始 markdown，图片引用保持原始相对路径
- `markdown_canonical`：保存重写后的 markdown，图片引用替换为 `asset://` URI，供内部检索使用

```go
type DocumentSource struct {
    Document          Document
    MarkdownOriginal  string     // 用户编辑时使用这个
    MarkdownCanonical string     // 内部检索使用这个
    Assets            []NodeAsset
}
```

新建版本时，编辑器加载 `markdown_original`，用户在熟悉的相对路径上编辑，提交时重新走 source package pipeline 解析。

### 影响范围

- `internal/store/postgres_migrate.go` — 新增两列
- `internal/kb/pg_store.go` — SaveDocument 写两列，GetDocumentSource 读 original
- `internal/kb/asset_ingest.go` — ingest 完成后回写 canonical
- Task 8 的实现逻辑需要调整

## P0-4：Batch Ingest 部分失败的数据一致性

### 源码事实

`pg_store.go` 中 `SaveDocument`（第 136 行）和 `SaveNodeAssets`（第 719 行）各自有独立事务，但两者之间没有跨文档的外层事务。

### 问题分析

计划 Task 7 Step 3 明确说：

> "如果失败发生在部分写入后，返回失败报告并保留成功文档；不假装原子性"

这个决策会带来以下问题：

1. **重试导致重复文档**：用户导入 4 篇文档，第 3 篇失败，前 2 篇已写入。用户重试整个包，前 2 篇会再次写入，出现重复 active 文档（唯一索引按 `content_hash + version`，如果内容相同会命中 ON CONFLICT，但如果用户修改了文档再重试则不会命中）。

2. **无法区分首次和重试**：日志中无法判断某个 batch 是首次导入还是重试，运维排查困难。

3. **用户体验混乱**：导入 4 篇，成功 2 篇，失败 1 篇，跳过 1 篇。用户不知道应该重试哪些，也不知道已成功的 2 篇是否需要手动清理。

### 修复方案

**方案 A（推荐）：全成功或全失败**

在 service 层用数据库事务包裹整个 batch：

```go
func (s *KBService) IngestPackage(ctx context.Context, input IngestPackageInput) (*IngestPackageResult, error) {
    // 1. 预检阶段（不写库）：所有文档通过预检才继续
    for _, doc := range selectedDocs {
        if err := s.precheck(doc); err != nil {
            return nil, fmt.Errorf("precheck failed at %s: %w", doc.Path, err)
        }
    }

    // 2. 写入阶段：用事务包裹
    tx, err := s.store.BeginTx(ctx)
    if err != nil {
        return nil, err
    }
    defer tx.Rollback()

    for _, doc := range selectedDocs {
        if err := s.ingestOne(ctx, tx, doc); err != nil {
            return nil, fmt.Errorf("ingest failed at %s: %w", doc.Path, err)
        }
    }
    return result, tx.Commit()
}
```

**方案 B（幂等性）**：如果跨文档事务实现成本高，至少实现幂等性——batch 请求携带客户端生成的 `idempotency_key`，服务端记录已成功的文档，重试时跳过已成功的，只重试失败的。

```go
type IngestPackageInput struct {
    IdempotencyKey string // 客户端生成，重试时保持不变
    // ...
}
```

### 影响范围

- `internal/kb/service.go` — IngestPackage 方法
- `internal/kb/store.go` — 可能需要暴露 BeginTx 接口
- `internal/kb/pg_store.go` — 跨文档事务实现
- API 契约（Section 3.2）— 可能增加 `idempotency_key` 字段
