## P1-1：内存炸弹 — 128MB 全加载到内存无流式处理

### 源码事实

`internal/api/kb_handlers.go` 中：

- 第 239 行：`http.MaxBytesReader` 限制请求体为 32MB
- 第 310 行：`r.ParseMultipartForm(maxKBIngestRequestBytes)` 将 multipart 数据解析到内存
- 第 1099-1109 行：`readMultipartFile` 使用 `io.ReadAll(file)` 将每个文件完整读入 `[]byte`

计划中提出将限制提升到 64MB zip / 128MB 解压后。

### 问题分析

**单请求内存占用估算**：

```
zip 文件本身:           64MB（请求体）
解压后文件内容:         128MB（SourcePackageFile.Data []byte）
SourcePackage 结构体:   ~1MB（元数据）
Preview Plan 响应体:    ~50MB（含 data URL base64 编码，膨胀 33%）
总计单请求峰值:         ~243MB
```

**并发场景**：

- 10 个管理员同时导入文档包 = 2.4GB 内存
- Go GC 在大对象上表现不佳，`[]byte` 超过 32KB 直接分配到堆上
- 容器内存限制通常 512MB-2GB，极易触发 OOM

**对比当前状态**：当前 32MB 限制下单请求峰值约 96MB，已经不小。提升到 64MB 后风险翻倍。

### 修复方案

**方案 A（推荐）：分阶段流式处理**

```go
type SourcePackageFile struct {
    Path     string
    Filename string
    Kind     SourcePackageFileKind
    MimeType string
    Size     int64
    // Data []byte  ← 删除，不再全量持有
    TempPath string // 临时文件路径，按需读取
}

func ParseZipSourcePackage(ctx context.Context, r io.ReaderAt, size int64, limits SourcePackageLimits) (*SourcePackage, error) {
    zr, err := zip.NewReader(r, size)
    // 第一遍：只读取文件列表和 metadata，不读取 body
    for _, f := range zr.File {
        pkg.Files = append(pkg.Files, SourcePackageFile{
            Path: f.Name,
            Size: int64(f.UncompressedSize64),
            // body 不读取
        })
    }
    // 第二遍：只读取 markdown 文件内容（通常很小）做图片引用扫描
    // 图片文件只在最终 ingest 时按需读取上传到对象存储
}
```

**方案 B（最小改动）：保持当前架构，严格限制并发**

```go
var packageIngestSemaphore = make(chan struct{}, 3) // 最多 3 个并发

func (s *Server) handleKBIngestPackage(w http.ResponseWriter, r *http.Request) {
    select {
    case packageIngestSemaphore <- struct{}{}:
        defer func() { <-packageIngestSemaphore }()
    default:
        http.Error(w, "too many concurrent package imports", http.StatusTooManyRequests)
        return
    }
    // ...
}
```

首发建议用方案 B 快速上线，方案 A 作为 P2 优化。

### 影响范围

- `internal/kb/source_package.go` — SourcePackageFile 结构体
- `internal/kb/source_package_zip.go` — 解析逻辑
- `internal/api/kb_handlers.go` — 并发控制

## P1-2：Zip Symlink 攻击未防御

### 源码事实

计划 Section 5.1 列出了 zip 安全防御清单：

- ✅ zip slip（`../`、绝对路径）
- ✅ zip bomb（膨胀比）
- ✅ 文件数过多
- ✅ 隐藏系统文件
- ✅ 可执行文件

**但没有提到 symlink。**

### 问题分析

Go 的 `archive/zip` 库支持读取 symlink 条目。攻击者可以构造：

```
docs/
  secret.md -> /etc/passwd          # 绝对路径 symlink
  config.md -> ../../../../.env     # 相对路径 symlink
```

如果代码在解压后 follow symlink 读取内容，会读到包外的敏感文件。

虽然当前计划是从 `[]byte` 直接解析（不落盘），symlink 在纯内存场景下不会被 follow。但如果未来改为临时文件落盘（如 P1-1 方案 A），symlink 就会成为真实攻击面。

### 修复方案

在 `ParseZipSourcePackage` 中增加检查：

```go
for _, f := range zr.File {
    // 检查 symlink
    if f.Mode()&os.ModeSymlink != 0 {
        pkg.Issues = append(pkg.Issues, SourcePackageIssue{
            Severity: "error",
            Code:     "symlink_not_allowed",
            Message:  fmt.Sprintf("symbolic link not allowed: %s", f.Name),
        })
        continue
    }

    // 检查文件类型（只允许普通文件和目录）
    if !f.Mode().IsRegular() && !f.Mode().IsDir() {
        pkg.Issues = append(pkg.Issues, SourcePackageIssue{
            Severity: "error",
            Code:     "unsupported_file_type",
            Message:  fmt.Sprintf("unsupported file type %s: %s", f.Mode().Type(), f.Name),
        })
        continue
    }
}
```

### 影响范围

- `internal/kb/source_package_zip.go` — 增加 3 行检查
- `internal/kb/source_package_test.go` — 增加 symlink 测试用例

## P1-3：并发导入同名文档的竞态条件

### 源码事实

`internal/kb/pg_store.go` 中 `SaveDocument`（第 136 行）的逻辑：

1. 开启事务（ReadCommitted）
2. `SELECT id FROM kb_documents WHERE namespace_id=$1 AND content_hash=$2 AND version=$3`
3. 如果存在，返回 `ErrDuplicateDocument`
4. 如果不存在，`INSERT ... ON CONFLICT (id) DO UPDATE`

数据库有唯一索引 `idx_kb_documents_namespace_hash_version`（`namespace_id, content_hash, version`，WHERE `content_hash != '' AND version != ''`）。

### 问题分析

**竞态窗口**：

```
T1: 请求 A — SELECT 检查 title="安装说明" 不存在 ✓
T2: 请求 B — SELECT 检查 title="安装说明" 不存在 ✓
T3: 请求 A — INSERT 成功
T4: 请求 B — INSERT 成功（content_hash 不同，唯一索引不拦截）
结果：两个 active 的"安装说明"文档
```

唯一索引是按 `(namespace_id, content_hash, version)` 建的，不是按 `(namespace_id, title, status)` 建的。所以两个不同内容但同名的文档可以同时 active。

计划中归档策略是"同 title 下旧 active 归档"，但归档发生在新文档写入之后。如果两个请求并发，都在归档前完成了写入，就会出现两个 active。

### 修复方案

**方案 A（推荐）：增加 partial unique index**

```sql
CREATE UNIQUE INDEX idx_kb_documents_active_title
ON kb_documents(namespace_id, domain_id, owner_scope, owner_id, title)
WHERE status = 'active';
```

这样数据库层面保证同一 namespace + owner + domain 下不会有两个同名 active 文档。第二个并发请求会收到唯一约束冲突，应用层捕获后返回 409。

**方案 B：应用层 advisory lock**

```go
func (s *PGStore) SaveDocumentWithLock(ctx context.Context, doc *Document) error {
    lockKey := hashString(doc.NamespaceID + doc.Title + doc.DomainID)
    _, err := s.db.ExecContext(ctx, "SELECT pg_advisory_xact_lock($1)", lockKey)
    if err != nil {
        return err
    }
    return s.SaveDocument(ctx, doc)
}
```

### 影响范围

- `internal/store/postgres_migrate.go` — 新增 partial unique index
- `internal/kb/pg_store.go` — 捕获唯一约束冲突，返回友好错误

## P1-4：前端大文档包 Preview 响应体过大导致卡顿

### 源码事实

计划 Section 3.1 的 preview 响应结构中：

- `documents[].markdown` — 每篇文档的完整 markdown 内容
- `assets[].data_url` — 每张图片的 base64 data URL

### 问题分析

**极端场景估算**：

```
200 篇 markdown × 平均 50KB = 10MB
100 张图片 × 平均 200KB × 1.33(base64) = 26.6MB
JSON 结构开销 = ~2MB
总响应体 ≈ 38.6MB
```

前端收到 38MB JSON 后：

1. `JSON.parse` 需要 ~2 秒（主线程阻塞）
2. React 渲染 200 个文档列表项 + 100 个缩略图 = 大量 DOM 节点
3. 浏览器内存占用 > 100MB（JSON 字符串 + 解析后对象 + DOM）
4. 低端设备直接卡死

### 修复方案

**分层加载**：

```typescript
// 第一层：只返回文件列表和 summary（< 10KB）
interface KBPackagePreviewSummary {
  package_id: string;
  summary: { total_files: number; markdown_files: number; errors: number; };
  documents: Array<{
    path: string;
    title: string;
    markdown_lines: number;
    issue_count: number;
    // 不含 markdown 正文
  }>;
  issues: KBPackageIssue[];
}

// 第二层：按需加载单篇文档预览
// GET /api/v1/kb/packages/{package_id}/documents/{path}/preview
interface KBPackageDocumentPreview {
  markdown: string;
  image_refs: MarkdownImageRef[];
}

// 第三层：按需加载单张图片缩略图
// GET /api/v1/kb/packages/{package_id}/assets/{path}/thumbnail
// 返回 image/jpeg，服务端压缩到 200px 宽
```

**最小改动方案**：保持当前 API 结构，但增加分页和可选字段：

```
POST /api/v1/kb/namespaces/{id}/documents:preview-package
  ?include_markdown=false    // 默认不返回正文
  &include_data_urls=false   // 默认不返回图片 data URL
  &page=1&page_size=20       // 文档列表分页
```

### 影响范围

- API 契约（Section 3.1）— 响应结构调整
- `internal/api/kb_handlers.go` — preview handler 支持分页/按需
- 前端 `KBPackagePreviewPanel.tsx` — 按需加载逻辑
