## 建议实施顺序

### 第一步：修复数据模型（在 Task 1 之前）

**目标**：确保数据模型正确，避免后续所有 Task 基于错误模型实现。

1. `kb_documents` 表增加字段：

```sql
ALTER TABLE kb_documents
  ADD COLUMN parent_document_id TEXT,
  ADD COLUMN lineage_root_id TEXT,
  ADD COLUMN markdown_original TEXT NOT NULL DEFAULT '',
  ADD COLUMN markdown_canonical TEXT NOT NULL DEFAULT '';

CREATE INDEX idx_kb_documents_lineage
  ON kb_documents(lineage_root_id, created_at);

CREATE UNIQUE INDEX idx_kb_documents_active_title
  ON kb_documents(namespace_id, domain_id, owner_scope, owner_id, title)
  WHERE status = 'active';
```

2. `Document` 结构体同步更新：

```go
type Document struct {
    // ... 现有字段 ...
    ParentDocumentID  *string `json:"parent_document_id,omitempty"`
    LineageRootID     *string `json:"lineage_root_id,omitempty"`
    MarkdownOriginal  string  `json:"-"`          // 不暴露给普通 API
    MarkdownCanonical string  `json:"-"`          // 不暴露给普通 API
}
```

3. 更新 `internal/kb/store.go` 接口，增加：

```go
GetDocumentSourceForManagement(ctx context.Context, scope ManagementScope, documentID string) (*DocumentSource, error)
ArchiveByLineage(ctx context.Context, lineageRootID string, excludeDocID string) error
```

**验收**：migration 跑通，现有测试不 break。

### 第二步：修复 Task 1-3（Source Package + Zip 解析）

在原计划 Task 1-3 基础上，额外增加：

1. **错误码常量化**（P2-3）：在 `source_package.go` 中定义 `IssueCode` 常量，所有 issue 通过工厂函数创建。

2. **Symlink 检查**（P1-2）：在 `source_package_zip.go` 的 `ParseZipSourcePackage` 中增加 `f.Mode()&os.ModeSymlink` 检查。

3. **并发限流**（P1-1 最小方案）：在 handler 层增加 semaphore，限制同时处理的 package 请求数。

```go
// internal/api/server.go
type Server struct {
    // ...
    packageSemaphore chan struct{} // 初始化为 cap=3
}
```

### 第三步：修复 Task 5（Preview Plan）

在原计划 Task 5 基础上，额外增加：

1. **Preview 缓存**（P0-2）：preview 结果写入内存缓存（TTL 5 分钟），返回 `package_id` 作为缓存 key。

2. **响应分层**（P1-4）：默认不返回 markdown 正文和 data URL，前端按需请求。

修改 API 契约：

```
POST /api/v1/kb/namespaces/{id}/documents:preview-package
  ?include_markdown=false
  &include_data_urls=false
```

增加按需接口：

```
GET /api/v1/kb/packages/{package_id}/documents/{index}/markdown
GET /api/v1/kb/packages/{package_id}/assets/{index}/thumbnail
```

---

### 第四步：修复 Task 7（Batch Ingest）

在原计划 Task 7 基础上，额外增加：

1. **复用 preview 缓存**（P0-2）：ingest 请求必须携带 `package_id`，从缓存读取已解析的 package，不重新上传。

2. **事务包裹**（P0-4）：整个 batch 在一个数据库事务内完成，全成功或全失败。

3. **幂等性兜底**：如果事务实现成本过高，至少增加 `idempotency_key` 字段，服务端记录已完成的 batch，重试时直接返回上次结果。

修改 API 契约（Section 3.2）：

```
POST /api/v1/kb/namespaces/{id}/documents:ingest-package
字段增加：
  package_id: string (必填，来自 preview 响应)
  idempotency_key: string (可选，客户端生成)
字段删除：
  package: file (不再重复上传)
  package_files: repeated file (不再重复上传)
```

---

### 第五步：修复 Task 8-9（Source 存储 + 新版本）

在原计划 Task 8-9 基础上，额外增加：

1. **双份 Markdown 存储**（P0-3）：ingest 时同时写入 `markdown_original`（原始）和 `markdown_canonical`（重写后）。

2. **版本链写入**（P0-1）：create-version 时设置 `parent_document_id` 和 `lineage_root_id`。

3. **归档逻辑改为按 lineage**（P0-1）：

```go
func (s *PGStore) ArchiveByLineage(ctx context.Context, lineageRootID, excludeDocID string) error {
    _, err := s.db.ExecContext(ctx, `
        UPDATE kb_documents
        SET status = 'archived', updated_at = NOW()
        WHERE lineage_root_id = $1 AND status = 'active' AND id <> $2
    `, lineageRootID, excludeDocID)
    return err
}
```

4. **首次导入的 lineage 处理**：首次导入时 `lineage_root_id = document.id`（自指），后续版本继承根文档的 `lineage_root_id`。

## 分期上线建议

### 第一期（MVP，建议 1 周）

**范围**：

- 小文档包导入（< 32MB，< 50 文件）
- 单线程导入，semaphore 限流
- 全成功或全失败（事务）
- 版本链完整
- 前端基础 UI

**限制**：

- 不提升 32MB 请求体限制
- 不做流式处理
- 不做批量数据库优化
- 前端不做虚拟滚动

**验收**：

```bash
go test ./internal/kb ./internal/api -count=1  # 全通过
# 手动测试：上传 10 个文件的 zip，preview → ingest → 新建版本 → 验证归档
```

### 第二期（生产级，建议 2 周）

**范围**：

- 提升到 64MB / 200 文件
- 流式处理大文件
- 批量数据库操作
- 并发上传图片到对象存储
- 完整 metrics 和告警
- 前端虚拟滚动 + 按需加载

**验收**：

```bash
# 性能测试：200 文件 zip 包，导入耗时 < 10s
# 压力测试：10 并发导入，无 OOM，无数据重复
# 监控验收：Grafana 面板可观测导入成功率、耗时分布
```

## 风险评估

| 修复项 | 引入新风险 | 缓解 |
|--------|-----------|------|
| 版本链字段 | 现有数据需要迁移（lineage_root_id 回填） | 迁移脚本：`UPDATE kb_documents SET lineage_root_id = id WHERE lineage_root_id IS NULL` |
| Preview 缓存 | 内存泄漏（缓存未清理） | TTL 5 分钟 + LRU 淘汰 + 最大 100 条 |
| 事务包裹 | 长事务锁表 | 200 文件 × 10ms/文档 = 2s 事务，可接受；超过 30s 自动超时 |
| Partial unique index | 现有数据可能违反约束 | 迁移前先查询并修复重复数据 |
| 双份 markdown | 存储空间翻倍 | 单篇 markdown 最大 2MB，200 篇 = 400MB，可接受 |

## 总结

这份计划的**产品设计和任务拆分质量很高**，路径解析纯函数化、不原地修改文档、安全限制全面等设计决策都是正确的。核心问题集中在**数据模型层**（版本链、markdown 存储）和**一致性保证**（preview/ingest 复用、事务原子性）。

建议先花 1-2 天修复数据模型设计，再按原计划推进实施。修复成本可控，但如果带着 P0 问题上线，后续数据迁移和修复的成本会高出一个数量级。
