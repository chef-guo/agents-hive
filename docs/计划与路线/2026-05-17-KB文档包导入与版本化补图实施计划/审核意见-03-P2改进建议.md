## P2-1：缺少 Metrics 监控 — 运维盲区

### 问题分析

计划 Section 6 定义了详细的日志字段（batch_id、duration_ms 等），但**只有日志没有 metrics**。日志是事后排查工具，metrics 是实时感知工具。

缺少 metrics 意味着：

- 无法设置告警（导入失败率突增、耗时突增）
- 无法做容量规划（每天导入量趋势、峰值并发）
- 无法发现慢退化（平均耗时从 200ms 逐渐涨到 2s）

### 建议方案

```go
import "github.com/prometheus/client_golang/prometheus"

var (
    kbPackageOpsTotal = prometheus.NewCounterVec(
        prometheus.CounterOpts{
            Name: "kb_package_ops_total",
            Help: "Total KB package operations",
        },
        []string{"operation", "mode", "status"},
        // operation: preview / ingest / create_version
        // mode: zip / folder / single
        // status: success / failed / rejected
    )

    kbPackageOpsDuration = prometheus.NewHistogramVec(
        prometheus.HistogramOpts{
            Name:    "kb_package_ops_duration_seconds",
            Help:    "KB package operation duration",
            Buckets: []float64{0.1, 0.5, 1, 2, 5, 10, 30},
        },
        []string{"operation", "mode"},
    )

    kbPackageFilesGauge = prometheus.NewHistogramVec(
        prometheus.HistogramOpts{
            Name:    "kb_package_files_count",
            Help:    "Number of files per package",
            Buckets: []float64{1, 5, 10, 20, 50, 100, 200},
        },
        []string{"file_type"}, // markdown / image / ignored
    )

    kbPackageRequestBytes = prometheus.NewHistogramVec(
        prometheus.HistogramOpts{
            Name:    "kb_package_request_bytes",
            Help:    "Request body size in bytes",
            Buckets: prometheus.ExponentialBuckets(1024, 4, 8), // 1KB ~ 64MB
        },
        []string{"mode"},
    )
)
```

**告警建议**：

| 指标 | 阈值 | 告警级别 |
|------|------|----------|
| 失败率 > 10%（5 分钟窗口） | P99 | Warning |
| 失败率 > 30%（5 分钟窗口） | P99 | Critical |
| P95 耗时 > 10s | 持续 5 分钟 | Warning |
| 并发导入 > 5 | 瞬时 | Info |

## P2-2：数据库批量操作未优化 — 性能瓶颈

### 问题分析

计划 Task 7 Step 3 的 service loop：

```go
for _, doc := range selectedDocs {
    s.IngestMarkdownWithAssets(ctx, doc) // 每篇文档独立调用
}
```

`IngestMarkdownWithAssets` 内部（基于 `pg_store.go` 源码）每篇文档至少涉及：

1. `INSERT INTO kb_documents` — 1 次
2. `INSERT INTO kb_tree_nodes` — N 次（按章节数，平均 5-10 次）
3. `INSERT INTO kb_node_assets` — M 次（按图片数）
4. 对象存储上传 — M 次

**200 篇文档 × 平均 8 个 tree node × 2 张图片 = 200 + 1600 + 400 = 2200 次数据库往返**

按每次 2ms 网络延迟计算：2200 × 2ms = 4.4 秒纯数据库等待。加上对象存储上传，总耗时可能 > 30 秒。

### 建议方案

**阶段一（首发可接受）**：保持当前逐条插入，但在事务内批量提交，减少事务开销。

**阶段二（后续优化）**：

```go
// 批量插入 tree nodes
func (s *PGStore) BatchInsertTreeNodes(ctx context.Context, tx pgx.Tx, nodes []TreeNode) error {
    _, err := tx.CopyFrom(ctx,
        pgx.Identifier{"kb_tree_nodes"},
        []string{"id", "document_id", "parent_id", "title", "text", "level", "page", "seq"},
        pgx.CopyFromSlice(len(nodes), func(i int) ([]interface{}, error) {
            n := nodes[i]
            return []interface{}{n.ID, n.DocumentID, n.ParentID, n.Title, n.Text, n.Level, n.Page, n.Seq}, nil
        }),
    )
    return err
}

// 并发上传图片到对象存储
func (s *KBService) uploadAssetsParallel(ctx context.Context, assets []AssetUpload) error {
    g, ctx := errgroup.WithContext(ctx)
    g.SetLimit(10) // 最多 10 并发
    for _, a := range assets {
        a := a
        g.Go(func() error {
            return s.assetStore.Upload(ctx, a.Key, a.Data, a.ContentType)
        })
    }
    return g.Wait()
}
```

**预期收益**：

- `CopyFrom` 比逐条 INSERT 快 5-10 倍
- 并发上传图片可将 M 次串行上传压缩到 ceil(M/10) 轮

## P2-3：错误码未标准化 — 维护成本高

### 问题分析

计划中 issue code 全部是硬编码字符串：

```go
issue.Code = "missing_asset"
issue.Code = "path_escapes_package"
issue.Code = "case_mismatch"
issue.Code = "reference_style_image_unsupported"
issue.Code = "remote_url_not_archived"
```

问题：

1. 拼写错误不会被编译器捕获
2. 前端需要硬编码同样的字符串做条件渲染
3. 没有统一的错误消息模板，同一个 code 在不同地方可能有不同 message
4. 国际化困难

### 建议方案

```go
// internal/kb/source_package_errors.go

type IssueCode string

const (
    IssueMissingAsset              IssueCode = "missing_asset"
    IssuePathEscapesPackage        IssueCode = "path_escapes_package"
    IssueAbsolutePath              IssueCode = "absolute_path"
    IssueCaseMismatch              IssueCode = "case_mismatch"
    IssueReferenceStyleUnsupported IssueCode = "reference_style_image_unsupported"
    IssueRemoteURLNotArchived      IssueCode = "remote_url_not_archived"
    IssueSymlinkNotAllowed         IssueCode = "symlink_not_allowed"
    IssueDuplicateCanonicalPath    IssueCode = "duplicate_canonical_path"
    IssueFileTooLarge              IssueCode = "file_too_large"
    IssueTooManyFiles              IssueCode = "too_many_files"
    IssuePathTooDeep               IssueCode = "path_too_deep"
)

// 消息模板，支持国际化
var issueTemplates = map[IssueCode]struct {
    Severity string
    ZH       string
    EN       string
}{
    IssueMissingAsset: {
        Severity: "error",
        ZH:       "找不到图片文件：%s（在 %s 第 %d 行引用）",
        EN:       "Image file not found: %s (referenced at %s line %d)",
    },
    IssuePathEscapesPackage: {
        Severity: "error",
        ZH:       "路径越界：%s 试图访问文档包外部",
        EN:       "Path escapes package boundary: %s",
    },
    // ...
}

func NewIssue(code IssueCode, args ...interface{}) SourcePackageIssue {
    tmpl := issueTemplates[code]
    return SourcePackageIssue{
        Severity: tmpl.Severity,
        Code:     string(code),
        Message:  fmt.Sprintf(tmpl.ZH, args...), // 根据 locale 选择
    }
}
```

前端对应：

```typescript
// frontend/src/constants/kb-issues.ts
export const KB_ISSUE_CODES = {
  missing_asset: { icon: 'error', color: 'red' },
  path_escapes_package: { icon: 'shield', color: 'red' },
  case_mismatch: { icon: 'warning', color: 'orange' },
  reference_style_image_unsupported: { icon: 'info', color: 'blue' },
} as const;
```

## P2-4：测试覆盖不足 — 缺少并发和性能测试

### 问题分析

计划中每个 Task 都有单元测试，覆盖了功能正确性。但缺少以下维度：

1. **并发测试**：多个 goroutine 同时导入同名文档
2. **性能测试**：1000 文件 zip 包的解析耗时
3. **压力测试**：连续导入 100 个包的内存增长曲线
4. **边界测试**：恰好在限制边界的文件（999 个文件、刚好 64MB）

### 建议补充

```go
// 并发安全测试
func TestConcurrentIngestSameTitle(t *testing.T) {
    store := setupTestStore(t)
    var wg sync.WaitGroup
    results := make([]error, 10)

    for i := 0; i < 10; i++ {
        i := i
        wg.Add(1)
        go func() {
            defer wg.Done()
            _, results[i] = store.IngestPackage(ctx, IngestPackageInput{
                NamespaceID: "ns-1",
                Package: buildSingleDocPackage("安装说明", fmt.Sprintf("content-%d", i)),
            })
        }()
    }
    wg.Wait()

    // 验证：只有一个成功，其余应该收到冲突错误
    successCount := 0
    for _, err := range results {
        if err == nil {
            successCount++
        }
    }
    assert.Equal(t, 1, successCount, "only one concurrent ingest should succeed")
}

// 性能基准测试
func BenchmarkParseZip1000Files(b *testing.B) {
    zipData := buildTestZipWithFiles(b, 1000) // 1000 个文件

            b.Fatal(err)
        }
    }
}

// 内存压力测试
func TestMemoryUsageUnderLoad(t *testing.T) {
    if testing.Short() {
        t.Skip("skipping memory test in short mode")
    }

    var m runtime.MemStats
    runtime.ReadMemStats(&m)
    baseAlloc := m.TotalAlloc

    for i := 0; i < 50; i++ {
        zipData := buildTestZipWithFiles(t, 100)
        _, _ = ParseZipSourcePackage(context.Background(), "test.zip", zipData, DefaultSourcePackageLimits())
    }

    runtime.GC()
    runtime.ReadMemStats(&m)
    growth := m.TotalAlloc - baseAlloc

    // 50 个包 × 100 文件，内存增长不应超过 500MB
    assert.Less(t, growth, uint64(500*1024*1024),
        "memory growth too high: %dMB", growth/1024/1024)
}
```
